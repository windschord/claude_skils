# SDD検証チェックリスト

SDDドキュメントをレビューする際の詳細な確認項目です。

## 全体チェック

- [ ] docs/ディレクトリ構造が存在する
- [ ] requirements/index.md、design/index.md、tasks/index.mdが存在する
- [ ] 3ドキュメント間の整合性が取れている
- [ ] 逆順レビューが完了している

## 要件定義書（docs/sdd/requirements/）

- [ ] ユーザーストーリーが明確に定義されている
- [ ] すべての要件がEARS記法に従っている
- [ ] 要件IDが一意である（REQ-XXX、NFR-XXX）
- [ ] 各要件がテスト可能である
- [ ] 非機能要件が含まれている

## 設計書（docs/sdd/design/）

- [ ] アーキテクチャ概要が記載されている
- [ ] 主要コンポーネントが定義されている
- [ ] インターフェースが明確である
- [ ] 技術的決定事項と根拠が記載されている
- [ ] 必要に応じて図表が含まれている
- [ ] 情報の明確性チェックが完了している
- [ ] 不明/要確認の情報がすべて解消されている
- [ ] requirements/の全要件に対応する設計要素がある
- [ ] CI/CD設計が含まれている（品質ゲート、GitHub Actions）
- [ ] 品質基準が定義されている（カバレッジ80%、ミューテーションスコア85%、Linter、複雑性）

## タスク管理書（docs/sdd/tasks/）

- [ ] タスクが適切な粒度に分解されている（20-40分程度）
- [ ] 各タスクに受入基準がある
- [ ] 依存関係が明確である
- [ ] 推定工数が記載されている
- [ ] ステータスが有効な値である（TODO/IN_PROGRESS/BLOCKED/REVIEW/DONE）
- [ ] 各タスクの「情報の明確性」セクションが記載されている
- [ ] 推測に基づく実装指示が含まれていない
- [ ] TDD手順が含まれている
- [ ] すべてのタスクがdesign/のコンポーネント/APIに対応している
- [ ] すべての設計要素がrequirements/に対応している

## 実装品質チェック

実装コードをレビューする際の品質観点です。全カテゴリを毎回確認する必要はなく、タスクの内容に応じて該当するカテゴリのみを選択して確認します。

### セキュリティ

- [ ] 外部入力（ユーザー入力、API引数、ファイルパス等）が適切にバリデーション・サニタイズされている
- [ ] パストラバーサル対策がされている（ファイルパスを扱う場合）
- [ ] コマンドインジェクション対策がされている（外部コマンドを実行する場合）
- [ ] 認証・認可が適切に実装されている（APIエンドポイントの場合）
- [ ] 機密情報（トークン、パスワード、環境変数の値等）がログやエラーメッセージに含まれていない
- [ ] デフォルト値が安全である（認証トークンのデフォルト、機能フラグ等）
- [ ] TOCTOU（Time-of-check to time-of-use）競合がない（検証と使用が分離している場合）
- [ ] 依存関係のバージョンが固定されている（latestタグ等の非固定参照がない）

### エラーハンドリング

- [ ] エラーが握りつぶされていない（catch節で空配列やnullを返して隠蔽していない）
- [ ] HTTPステータスコードが適切に使い分けられている（入力エラーが500で返らない等）
- [ ] エラーの種類に応じた区別がされている（Not Found、バリデーションエラー、サイズ超過等）
- [ ] 例外発生時にリソースリークがない（タイマー、コネクション、ファイルハンドル等が適切にクリーンアップされる）
- [ ] undefinedやnullの安全な取り扱いがされている（スプレッド演算子やプロパティアクセス時）
- [ ] エラーログの形式が既存コードのパターンと統一されている（ログレベル、メッセージ構造、コンテキスト情報の含め方）

### インターフェース整合性

- [ ] フロントエンドとバックエンドでリクエスト/レスポンスのフィールド名が一致している
- [ ] 一覧API・詳細APIなど関連するエンドポイント間でポリシー（フィルタ、権限等）が一貫している
- [ ] 設定キー名の命名規則（camelCase/snake_case等）がレイヤー間で統一されている
- [ ] コメント・ドキュメントの記述と実際の実装の挙動が一致している

### 後方互換性

- [ ] 既存の公開APIやデータ形式に破壊的変更がないか確認している
- [ ] 破壊的変更がある場合、マイグレーション手順が用意されている
- [ ] レガシーデータ・既存セッション等が新バージョンで孤児化しない
- [ ] 設定値の変更・廃止時にバリデーションやフォールバックが実装されている

### テスト品質

- [ ] 新規コンポーネント・機能に対応するテストが追加されている
- [ ] モックの振る舞いが実際の実装と一致している（呼び出し回数、引数、戻り値）
- [ ] テストが環境に依存していない（実コマンド実行、固定パス、特定OS前提等がない）
- [ ] テスト名・テストの意図と検証内容が一致している
- [ ] エラーケースのテストが含まれている

### パフォーマンス

- [ ] N+1クエリ・N+1リクエストが発生していない（ループ内での個別API呼び出し等）
- [ ] 不要な再計算がない（メモ化すべき計算が毎回実行されていない）
- [ ] バッファサイズ・ファイルサイズの上限が考慮されている
- [ ] ディレクトリ探索やファイル走査に適切な範囲制限がある

### コード品質

- [ ] 未使用のimport・変数・関数がない
- [ ] コメントと実装が乖離していない（コメントでは「Aする」と書いて実際はBしている等）
- [ ] 命名規則が既存コードのパターンと統一されている（変数・関数・クラス・ファイル名）
- [ ] モジュールのimport方式が既存コードと統一されている（ESM/CommonJSの混在がない等）

## 逆順レビュー

### タスク → 設計の整合性

| チェック項目 | 確認内容 |
|-------------|---------|
| コンポーネント対応 | タスクが参照するコンポーネントはdesign/components/に定義されているか |
| API対応 | タスクで実装するAPIはdesign/api/と一致しているか |
| データモデル対応 | タスクで使用するデータ構造はdesign/database/と一致しているか |
| 技術スタック | タスクで使用する技術はdesign/decisions/と一致しているか |

### 設計 → 要件の整合性

| チェック項目 | 確認内容 |
|-------------|---------|
| 機能カバレッジ | すべての要件（REQ-XXX）に対応する設計要素があるか |
| 非機能要件対応 | NFR-XXXの要件が設計に反映されているか |
| 過剰設計チェック | requirements/にない機能が設計に含まれていないか |

## 不整合発見時の報告テンプレート

```text
ドキュメントの整合性チェックで以下の不整合を発見しました：

【タスク → 設計の不整合】
1. tasks/phase-2/TASK-003.mdで「PaymentService」を実装するとありますが、
   design/components/にPaymentServiceの定義がありません。

【設計 → 要件の不整合】
2. design/components/に「通知機能」がありますが、
   requirements/stories/に対応する要件がありません。

【過不足】
3. REQ-005（レポート出力機能）に対応するタスクがありません。

これらについて確認させてください：
1. PaymentServiceの設計を追加しますか？
2. 通知機能は必要ですか？
3. REQ-005のタスクを追加しますか？
```
