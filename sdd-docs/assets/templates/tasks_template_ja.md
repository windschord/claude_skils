## タスクステータスの凡例

- `BLOCKED` - 依存関係や問題によりブロック中
- `REVIEW` - レビュー待ち
- `DONE` - 完了

## タスク管理のガイド

### タスクの粒度
適切なタスクサイズの目安：
- **小タスク**: 2-4時間（半日以内）
- **中タスク**: 4-8時間（1日）
- **大タスク**: 8-16時間（2-3日） → さらに分解を検討

### 良いタスク定義の特徴
1. **具体的**: 何をするか明確
2. **測定可能**: 完了基準が明確
3. **達成可能**: 現実的な工数
4. **関連性**: プロジェクト目標に貢献
5. **期限付き**: 推定工数が明記

### 受入基準の書き方
- チェックボックス形式で記載
- 客観的に判定可能な基準
- テスト可能な条件
- ユーザー視点での成果物

### 依存関係の管理
- **前提タスク**: このタスクの開始前に完了が必要
- **並行可能**: 同時に進められるタスク
- **後続タスク**: このタスクの完了後に開始可能

### ステータス遷移のルール
```
TODO → IN_PROGRESS → REVIEW → DONE
         ↓
      BLOCKED
         ↓
    IN_PROGRESS
```

### フェーズ分けの考え方
1. **基盤構築**: 環境設定、基本構造
2. **コア機能**: 主要な機能の実装
3. **拡張機能**: 追加機能、改善
4. **統合・テスト**: 結合、品質保証
5. **リリース準備**: デプロイ、文書化

### リスク評価のマトリクス
|  | 低影響 | 中影響 | 高影響 |
|---|--------|--------|--------|
| **高確率** | 注意 | 対策必要 | 最優先対応 |
| **中確率** | 監視 | 注意 | 対策必要 |
| **低確率** | 受容 | 監視 | 注意 |

### 推定工数のバッファ
- 楽観的見積もり × 1.5 = 現実的見積もり
- 不確実性が高い場合は × 2.0
- 新技術や未経験領域は × 2.5-3.0

### 進捗管理のヒント
- 毎日ステータスを更新
- ブロッカーは即座に報告
- 見積もりと実績の差を記録
- 振り返りで改善点を特定
