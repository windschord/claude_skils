## 2. SLA（Service Level Agreement）

### 2.1 サービスレベル目標

**可用性・稼働率**

| 項目 | 目標値 | 測定方法 | 除外事項 |
|------|--------|----------|----------|
| サービス稼働率 | 99.9% | 月次での稼働時間/総時間 | 計画メンテナンス、顧客起因の停止 |
| API可用性 | 99.95% | API応答成功率 | DDoS攻撃などの外部要因 |

**記入例**
```
- サービス稼働率: 99.9%以上（月次計測）
  - 計算式: (総時間 - ダウンタイム) / 総時間 × 100
  - 除外: 事前通知した計画メンテナンス（月次1回、最大4時間）
```

**説明**
- 99.9%の可用性 = 月間約43分のダウンタイムまで許容
- 99.95%の可用性 = 月間約22分のダウンタイムまで許容
- 目標値はビジネス要件とコストのバランスで決定

**計画メンテナンス**

| 項目 | 内容 |
|------|------|
| メンテナンス時間 | [曜日] [時間帯] |
| 通知期間 | メンテナンス実施の[X]日前 |
| 最大停止時間 | [X]時間 |

**記入例**
```
- 定期メンテナンス: 毎月第2日曜日 02:00-04:00 JST
- 緊急メンテナンス: 重大な脆弱性対応など、必要に応じて実施
- 通知: 計画メンテナンスは7日前、緊急は可能な限り事前通知
```

### 2.2 性能・レスポンス

**パフォーマンス目標**

| 項目 | 目標値 | 測定条件 | 対応方針 |
|------|--------|----------|----------|
| API平均レスポンス | 200ms以下 | P95パーセンタイル | 閾値超過時はアラート発報 |
| ページロード時間 | 2秒以下 | フロントエンド初期表示 | CDN、キャッシュ最適化 |
| データベースクエリ | 100ms以下 | 主要クエリのP99 | スロークエリの継続監視 |

**記入例**
```
- API平均レスポンスタイム: 200ms以下（95パーセンタイル）
  - 測定: 1分間隔での計測、5分間の移動平均
  - 閾値超過時: 自動アラート発報、原因調査開始
```

**説明**
- P95パーセンタイル: 全リクエストの95%がこの時間内に完了
- P99パーセンタイル: より厳しい基準、99%が指定時間内に完了

### 2.3 バックアップ・データ保護

**RTO/RPO定義**

| データ種別 | RPO | RTO | バックアップ頻度 |
|------------|-----|-----|------------------|
| 本番データベース | 15分 | 4時間 | 15分ごとの増分 + 日次フル |
| ユーザーファイル | 1時間 | 8時間 | 1時間ごとのスナップショット |
| システム設定 | 24時間 | 2時間 | 日次バックアップ |

**用語説明**
- RPO (Recovery Point Objective): データ損失許容時間。「最悪どこまでデータが失われてもよいか」
- RTO (Recovery Time Objective): 復旧目標時間。「どれくらいの時間で復旧すべきか」

**記入例**
```
- 本番データベース:
  - RPO: 15分（最大15分前の状態まで復旧可能）
  - RTO: 4時間（障害発生から4時間以内にサービス復旧）
  - バックアップ: 15分ごとの増分バックアップ、毎日03:00にフルバックアップ
```

**バックアップ保管**

| 項目 | 内容 |
|------|------|
| 保管場所 | [1次保管場所、2次保管場所] |
| 保管期間 | [日次: X日、週次: X週、月次: Xヶ月] |
| 暗号化 | [暗号化方式] |
| リストアテスト | [頻度: 月次/四半期] |

**記入例**
```
- 1次保管: 同一リージョン内の別アベイラビリティゾーン
- 2次保管: 別リージョン（地理的に離れた場所）
- 保管期間:
  - 日次バックアップ: 30日間
  - 週次バックアップ: 12週間
  - 月次バックアップ: 12ヶ月間
- 暗号化: AES-256による暗号化
- リストアテスト: 四半期ごとに実施、手順書の有効性確認
```

### 2.4 インシデント対応時間

**対応時間の定義**

| 障害レベル | 初動対応 | 暫定対応 | 恒久対応 | 報告 |
|-----------|----------|----------|----------|------|
| Critical (Level 1) | 15分以内 | 2時間以内 | 24時間以内 | 即座 + 事後詳細 |
| High (Level 2) | 30分以内 | 4時間以内 | 48時間以内 | 4時間以内 |
| Medium (Level 3) | 2時間以内 | 8時間以内 | 5営業日以内 | 翌営業日 |
| Low (Level 4) | 4時間以内 | 2営業日以内 | 2週間以内 | 週次レポート |

**障害レベルの定義**

**Level 1 (Critical): サービス全停止、大規模障害**
- 全ユーザーがサービスを利用できない
- データ損失の可能性がある
- セキュリティインシデント（情報漏洩など）
- 記入例: 本番環境のデータベース停止、全APIの応答不能

**Level 2 (High): 重要機能停止、大幅な性能劣化**
- 主要機能の一部が利用できない
- 複数のテナントに影響
- パフォーマンスが通常の50%以下
- 記入例: ログイン機能の停止、ファイルアップロード機能の停止

**Level 3 (Medium): 軽微な障害、代替手段あり**
- 一部機能の不具合だが、回避策がある
- 特定のテナントのみに影響
- 記入例: レポート出力の遅延、一部UIの表示崩れ

**Level 4 (Low): 問い合わせ、改善要望**
- サービスの提供に影響なし
- 記入例: ドキュメントの誤記、軽微なUI改善要望

**バッチジョブ失敗時の対応レベル**

バッチ処理の失敗は、ジョブの種類と影響範囲により以下のように分類します。

| ジョブ重要度 | 対応レベル | 初動対応 | 再実行 | 通知先 |
|-------------|-----------|----------|--------|--------|
| Critical | Level 1 | 15分以内 | 即座に手動再実行 | 運用チーム + 開発チーム |
| High | Level 2 | 30分以内 | 2時間以内に再実行 | 運用チーム |
| Medium | Level 3 | 2時間以内 | 翌営業日までに再実行 | 運用チーム |
| Low | Level 4 | 翌営業日 | 次回スケジュール実行時 | ログのみ |

**記入例**
```
Critical（緊急バッチジョブ）:
- 課金処理バッチ
- データ同期バッチ（外部システムとのSLA契約あり）
- セキュリティ関連バッチ（アクセスログ集計等）
- 対応: 失敗検知から15分以内に原因調査、手動再実行

High（重要バッチジョブ）:
- 日次レポート生成
- ユーザー通知バッチ
- データバックアップ
- 対応: 営業時間内に対応、2時間以内に再実行

Medium（通常バッチジョブ）:
- 統計データ集計
- ログアーカイブ
- 対応: 翌営業日までに対応

Low（低優先度バッチジョブ）:
- データクリーンアップ
- 一時ファイル削除
- 対応: 次回スケジュール実行時に自動再実行
```

### 3.3 IAM/アクセス制御設計

**サービスアカウントと権限設計**

コンテナ実行基盤では、コンテナの起動用とアプリケーション実行用の2種類の権限設定が必要です。

| 権限種別 | 用途 | 権限範囲 | 使用タイミング |
|----------|------|----------|----------------|
| 実行基盤用権限 | コンテナの起動に必要な操作 | レジストリ、ログ、シークレット管理 | コンテナ起動時 |
| アプリケーション用権限 | アプリケーションコードが実行する操作 | ストレージ、DB、キュー等 | コンテナ実行中 |

**最小権限の原則**

**記入例**
```
権限設計原則:

1. 最小権限の原則:
   - 必要な権限のみを付与
   - ワイルドカードの使用は最小限に
   - リソースを明示的に指定

2. 環境ごとの分離:
   - 本番環境用: prod-app-service-role
   - ステージング環境用: stg-app-service-role
   - 開発環境用: dev-app-service-role

3. サービスごとの分離:
   - APIサービス: api-service-role
   - ワーカーサービス: worker-service-role
   - バッチ処理: batch-service-role

4. 定期的な監査:
   - 未使用権限の検出
   - 月次でのロール使用状況レビュー
   - 過剰な権限の削除
```

### 3.4 コスト最適化戦略

**コンピュートリソースの選択**

| 実行方式 | 特徴 | コスト | 中断リスク | 用途 |
|----------|------|--------|------------|------|
| オンデマンド/標準 | 標準のコンピュートリソース | 標準 | なし | 本番の重要ワークロード |
| スポット/プリエンプティブル | 余剰リソースの活用 | 最大70-90%削減 | あり（事前通知あり） | 開発環境、中断許容バッチ |

**記入例**
```

本番環境:
- ベースライン: オンデマンド/標準（最小レプリカ数を確保）
- スケールアウト時: スポット/プリエンプティブルを30%混合（コスト削減）

ステージング環境:
- 全てスポット/プリエンプティブルを使用（コスト最優先）

開発環境:
- 全てスポット/プリエンプティブルを使用

バッチ処理:
- スポット/プリエンプティブルを使用
- ジョブは中断時に再実行可能な設計
- グレースフルシャットダウン対応
```

**スポットインスタンス中断時の対応**

[中断通知を受けた際のハンドリング方法を記載]

**記入例**
```
スポット中断対応:

1. 中断通知の監視:
   - イベント通知サービスで中断通知を検知
   - 通常2分前に通知を受信

2. グレースフルシャットダウン:
   - SIGTERMシグナルを受信
   - 実行中のリクエスト処理を完了
   - 新規リクエストは受け付けない
   - 最大120秒でシャットダウン

3. 自動リカバリ:
   - オーケストレーターが自動的に新しいインスタンスを起動
   - サービスの希望レプリカ数を維持

4. アプリケーション設計:
   - ステートレス設計（再起動に耐えられる）
   - 未完了処理はキューに保存
   - 冪等性の確保（同じ処理を複数回実行しても問題ない）
```

### 3.6 ステートレス設計とデータ永続化

**ステートレスアーキテクチャの原則**

コンテナはいつでも停止・再起動される可能性があるため、ステートレス設計が必須です。

**記入例**
```
ステートレス設計原則:

1. コンテナ内にデータを保存しない:
   - ローカルファイルシステムは揮発性
   - コンテナ再起動でデータ消失

2. セッション管理:
   - セッション情報はRedis等の外部ストアに保存
   - スティッキーセッションは使用しない
   - JWTトークンでステートレス認証

3. ファイル処理:
   - アップロードファイルは即座にオブジェクトストレージへ保存
   - 一時ファイルは処理後に削除
   - 大容量ファイルはプリサインドURL経由

4. ログ:
   - 集中ログ管理サービスへストリーミング
   - ローカルログファイルは保持しない

5. キャッシュ:
   - マネージドRedis/Memcachedを使用
   - コンテナ内メモリキャッシュは最小限
```

**データ永続化の選択肢**

| 方式 | 用途 | パフォーマンス | コスト | 制約 |

| 共有ファイルシステム | 複数コンテナ間のファイル共有 | 中 | 中 | レイテンシに注意 |
| マネージドRDB | リレーショナルデータ | 高 | 高 | トランザクション |

**記入例**
```

- データベース:

  - テーブルレベルでの追加暗号化（機密データ）

- カスタマーマネージドキーを使用
- キーローテーション: 自動で年1回
- キーへのアクセスはIAMポリシーで厳格に制御
```

|----------|----------|------|--------------|

| 2. トリアージ | 影響範囲の特定、優先度判定 | オンコール担当 | 15分 |
| 3. 対応 | 原因調査、暫定対応、恒久対応 | 運用チーム | 障害レベルによる |
| 4. 復旧 | サービス正常化、動作確認 | 運用チーム | 障害レベルによる |

**記入例**
```

  - 外形監視からの通知
  - ユーザーからのサポート問い合わせ
  - 内部スタッフからの報告

  - チャットでインシデントチャンネル作成
  - ステータスページの更新（Investigating）

フェーズ5: 事後対応（3営業日以内）
- インシデントレポート作成
- ポストモーテム（Blameless - 責任追及ではなく改善重視）
```

## 12. 変更管理・リリース管理

### 12.1 変更管理プロセス

**変更分類**

| 変更分類 | 定義 | 承認者 | リードタイム |
|----------|------|--------|--------------|
| 標準変更 | 手順化済みの低リスク変更 | 自動承認 | 即時 |
| 通常変更 | 一般的な変更 | チームリーダー | 1営業日 |
| 重要変更 | 高リスクまたは大規模な変更 | 運用責任者 | 3営業日 |
| 緊急変更 | 緊急対応が必要な変更 | 運用責任者（事後承認可） | 即時 |

**記入例**
```
標準変更:
- CI/CDパイプライン経由のアプリケーションデプロイ
- 環境変数の追加・変更（非機密情報）
- 監視閾値の調整
- 自動承認、即座に実施可能

通常変更:
- データベースのスキーマ変更
- インフラリソースの追加・変更
- セキュリティグループ/ファイアウォールルール変更
- チームリーダーの承認後に実施

重要変更:
- データベースの大規模移行
- メジャーバージョンアップグレード
- ネットワーク構成の変更
- 運用責任者の承認後に実施

緊急変更:
- セキュリティ脆弱性のパッチ適用
- 障害対応のための設定変更
- 事後承認可、速やかに実施
```

**デプロイメント戦略**

| 方式 | メリット | デメリット | 採用判断 |
|------|----------|------------|----------|
| Blue-Green | ダウンタイムゼロ、即時ロールバック | リソース2倍必要 | [○/×] |
| Rolling | リソース効率的 | 段階的な切り替え | [○/×] |

### 14.3 変更管理

本ドキュメントの変更は以下のプロセスに従います:

1. 変更提案（プルリクエスト）
2. レビュー（運用責任者 + 該当チームリーダー）
3. 承認
4. マージ・公開
5. 関係者への通知

**見直しスケジュール**
- 四半期ごとの定期見直し
- 重大インシデント後の臨時見直し
- システムアーキテクチャ変更時の臨時見直し

## 運用設計書作成時の注意事項

本テンプレートを使用する際は、以下の点に注意してください:

1. **クラウドプロバイダに応じたカスタマイズ**
   - テンプレート内の汎用的な記述をご利用のクラウドプロバイダ（AWS / GCP / Azure等）の具体的なサービス名に置き換えてください
   - プロバイダ固有の機能や制限を反映してください

2. **プロジェクト特性に応じた取捨選択**
   - すべてのセクションが必要とは限りません
   - プロジェクトの規模や性質に応じて取捨選択してください

3. **具体性の確保**
   - 抽象的な記述ではなく、具体的な数値や手順を記載してください
   - 「適切に」「十分に」などの曖昧な表現は避けてください

4. **測定可能性**
   - 目標値は測定可能な形で定義してください
   - 測定方法も明記してください

5. **継続的な更新**
   - 運用開始後も定期的に見直し、更新してください
   - 実態との乖離が生じないよう注意してください

6. **ステークホルダーの合意**
   - 関係者全員で内容を確認し、合意を形成してください
   - 特にSLOや運用体制については十分な議論が必要です
