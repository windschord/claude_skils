# 優先度分類基準（後続作業への影響度）

顧客レビュー用Excelと運用設計書付録で提示する優先度（高・中・低）の分類基準を定義する。分類の実体は項目マスタ `assets/master/ipa_nfr_items_ja.csv` の `priority` 列であり、本書はその判断基準と調整方法を示す。

## 分類の目的

顧客が限られたレビュー時間で「先に決めないと後戻りコストが大きい項目」から確認できるようにするため、**後続の設計・構築・運用準備作業への影響度**で全項目を3段階に事前分類している。

## 分類基準

### 高（32項目）: 後続作業の根幹を決める項目

以下のいずれかに該当する項目:

1. **インフラ構成・アーキテクチャを直接決定する**
   - 例: 稼働率(A.2.1.1)、冗長構成(A.1.1.2)、スケール方式(B.3.1.1)、HW専有要否(B.4.1.2)、稼働環境(F.1.1.1)
2. **変更時の後戻りコストが極めて大きい**
   - 例: 認証方式(E.1.1.1)、MFA要否(E.1.1.2)、通信・保存データ暗号化(E.2.1.x)
3. **他の多数の設計項目の前提となる**
   - 例: サービス稼働時間(A.1.2.1)、RTO/RPO/RLO(A.1.3.x)、業務量・応答時間(B.1.x)、データ容量(B.2.2.1)
4. **法規制・契約に関わり必須要件を規定する**
   - 例: 適用法規制(E.5.1.1)、ログ保管期間(C.5.1.2)
5. **プロジェクト全体のクリティカルパスを決める**
   - 例: 移行期限(D.1.1.1)、移行方式(D.1.1.2)、移行中の許容停止時間(D.1.1.3)

**レビュー観点**: これらが未確定・誤っていると設計のやり直しが発生するため、顧客レビューで最優先に確定させる。

### 中（44項目）: 運用プロセス・手順の設計に影響する項目

- 運用プロセスの統制レベルを決める（ITILプロセス C.6.x、変更・パッチ管理 C.4.x）
- 運用スケジュール・手順の制約となる（計画停止 A.2.2.x/A.4.1.1、ジョブ管理 C.3.2.x）
- 上位の「高」項目に従属して方式が決まる（バックアップ世代数 C.2.1.2 は取得間隔 C.1.2.5 に従属、等）

**レビュー観点**: 設計工程内で調整可能だが、早期確定が望ましい。

### 低（12項目）: 付加的・詳細レベルの項目

- 成果物・体制の付加要素（ヘルプデスク A.2.6.1/C.1.2.7、ドキュメント整備 A.2.6.2/C.1.2.1）
- ライフサイクル終期・中長期の要件（機器更改 A.2.5.3、HW廃棄 F.2.2.1）
- 手順書レベルの詳細（リラン方針 C.3.2.3、パッチ適用後テスト範囲 C.4.1.3）

**レビュー観点**: 後工程での調整が容易。レビュー時間が限られる場合は高・中を優先してよい。

## プロジェクト固有の調整

業種・システム特性によっては標準分類が適さない場合がある（例: 金融・医療では監査ログE.4.xを「高」に引き上げる、バッチ中心システムではジョブ関連C.3.2.xを「高」に引き上げる）。

調整する場合の手順:

1. マスタCSVをプロジェクト用にコピーし、`priority` と `priority_reason` を修正する
2. `init --master <プロジェクト用CSV>` でDBを初期化する
3. 調整内容（項目・変更前後・理由）をユーザーに提示して承認を得る

**注意**: 初期化済みDBの優先度を変更する場合は、DBを再初期化するとselections/feedbackが失われるため、sqlite3で `UPDATE nfr_items SET priority = ?, priority_reason = ? WHERE item_id = ?` を直接実行すること。

## Excel・Wordでの表現

| 成果物 | 優先度の表現 |
|--------|-------------|
| Excel「非機能要求グレード」シート | 高→中→低の順に整列。優先度列を色分け（高: 赤系 / 中: 黄系 / 低: 灰色） |
| Excel「記入ガイド」シート | 各優先度の意味と推奨レビュー順を説明 |
| Word付録「非機能要件選択一覧」 | 優先度別のサブセクションに分けて表を掲載 |
