# 分析の5大原則

ヘルスチェックにおけるメトリクス分析の基本原則を解説します。

---

## 原則1: 周期性を意識する

あらゆるスパイクを見つけたら「これは周期的か、突発的か」を判断します。

### 周期パターン一覧

| 周期 | 例 | 確認方法 |
|------|---|---------|
| 日次 | 深夜バッチ、営業時間帯トラフィック | 過去7日間の同時刻と比較 |
| 週次 | 平日/週末差、週次バッチ | 前週同曜日と比較 |
| 月次 | 月初請求、月末締め | 前月同日と比較 |
| 商業イベント | セール、キャンペーン、新商品発売 | イベントカレンダーと照合 |

### 判断フロー

```text
スパイクを検出
  ├── ナレッジベースに既知パターンとして記録されている
  │     → 周期的: 正常範囲として報告
  ├── 過去データと比較して同時刻/同曜日に同様のパターンがある
  │     → 周期的の可能性高い: 確認の上ナレッジベースに追加
  └── 過去に類似パターンがない
        → 突発的: 異常として報告し原因調査
```

### 注意事項

- 周期的なスパイクでも、過去と比較して**振幅が大幅に増大**している場合は注意
- 商業イベントによるスパイクは事前情報がないと周期的か判断できない
- 新しい定期バッチが追加された場合、初回は突発的に見えるため確認が必要

## 原則2: 相関分析を行う

単一メトリクスではなく、時間的に一致する複数メトリクスの組み合わせで判断します。

### 主要な相関パターン

#### パターン1: インフラ障害
```text
アラート発火 + リクエスト変動 + コスト変動
→ サービス影響の可能性あり
→ アクション: 障害対応フローに移行
```

#### パターン2: リソース逼迫
```text
CPU上昇 + レスポンスタイム悪化 + エラー増
→ リソース不足によるサービス劣化
→ アクション: スケーリング検討、ボトルネック調査
```

#### パターン3: JVMヒープ枯渇
```text
GC頻発 + GCポーズ増大 + p95悪化
→ JVMヒープメモリの枯渇
→ アクション: ヒープダンプ取得、メモリリーク調査
```

#### パターン4: メッセージ処理障害
```text
キュー滞留 + DLQメッセージ増加
→ メッセージ処理の障害
→ アクション: コンシューマのログ確認、処理失敗原因の調査
```

#### パターン5: 正常スケーリング
```text
コスト増 + リクエスト増（連動）
→ 需要増に伴う正常なスケーリング
→ アクション: 経過観察、キャパシティプランニング
```

#### パターン6: インフラ変更起因
```text
コスト増 + リクエスト横ばい（乖離）
→ インフラ変更による影響
→ アクション: 変更内容の確認、意図した変更かの確認
```

#### パターン7: キャッシュ破壊
```text
Eviction増加 + DB CPU上昇 + レスポンスタイム悪化
→ キャッシュ効果の低下によるバックエンド負荷増
→ アクション: キャッシュサーバーのメモリ増設、キャッシュ戦略見直し
```

#### パターン8: ストレージ逼迫
```text
ディスク使用率上昇 + DB CPU上昇 + レスポンスタイム悪化
→ ストレージI/Oのボトルネック
→ アクション: 不要データのクリーンアップ、ストレージ拡張
```

### 相関分析の手順

1. 異常を検出したメトリクスの**発生時刻**を特定する
2. 同じ時間帯に変動している**他のメトリクス**を確認する
3. 上記のパターンに当てはめて**状況を推定**する
4. パターンに当てはまらない場合は**個別に調査**する
5. 相関が**なかった場合も記録**する（調査の網羅性を示す）

## 原則3: アカウント/テナント間の依存関係を把握する

共通基盤を使うサービス間では、一方の障害が他方に波及します。

### 依存関係の種類

| 依存関係 | 例 | 波及パターン |
|----------|---|-------------|
| 認証基盤 | SSO、OAuth、IAM | 認証障害で全サービス影響 |
| 共通データベース | マスタDB、共有スキーマ | DB負荷で依存サービス全体が遅延 |
| 共通キャッシュ | 共有Redis、セッションストア | キャッシュ障害でセッション消失 |
| ネットワーク | VPC、VPN、Transit Gateway | ネットワーク障害で複数サービス影響 |
| 外部API | 決済、メール、SMS | 外部サービス障害でAPI遅延 |

### 確認の手順

1. 異常が複数サービスで同時に発生している場合、**共通コンポーネント**を確認
2. ナレッジベースの**依存関係マップ**を参照
3. 依存先が別の監視グループに属する場合は**見落としやすい**ため注意
4. 波及元のサービスが特定できたら、そのサービスの詳細調査を優先

## 原則4: 閾値の方向性を理解する

メトリクスによって「高い方が正常」「低い方が正常」が異なります。

### パターン分類

| パターン | メトリクスの例 | 判定ルール |
|----------|---------------|-----------|
| 低い方が正常 | CPU使用率、エラー率、ディスク使用率 | 値が上昇したら注意 |
| 高い方が正常 | キャッシュメモリ余裕度、CPUクレジット残高、SSL残日数 | 値が低下したら注意 |
| ゼロが正常 | スロットリング、デッドロック、5xx、Unhealthy host、Eviction | ゼロ以外は即時確認 |

### よくある誤判定

- **キャッシュメモリ余裕度**: 「高い=異常」と誤判定しがち。**高い方が正常**
- **CPUクレジット残高**: 「高い=異常」と誤判定しがち。**高い方が正常**（枯渇が問題）
- **SSL証明書残日数**: 残日数の**少ない方が危険**。自動更新サイクル内かどうかの判断も必要

### レポートでの扱い

- 閾値方向が直感と逆のメトリクスは、レポートで**正常である理由を補足**する
- 例: 「キャッシュメモリ余裕度: 75%（正常。このメトリクスは高い方が正常）」

## 原則5: 事前情報を収集してから調査する

メトリクスの値だけでなく、コンテキスト情報を加味して判断します。

### 収集すべき事前情報

| カテゴリ | 内容 | 確認方法 |
|----------|------|---------|
| インフラ変更 | デプロイ、スケーリング、バージョンアップ | 変更管理ツール、デプロイログ |
| エスカレーション | 対応中の案件の経過 | チケットシステム、前回レポート |
| 商業イベント | セール、キャンペーン、新商品発売 | 事業部門への確認、カレンダー |
| クラウド障害 | AWS/GCP/Azure等の障害情報 | ステータスページ、SNS |

### 事前情報の活用

- **デプロイ直後のメトリクス変動**: デプロイ内容との関連を確認
- **スケーリング後のコスト増**: 意図した変更であれば正常
- **商業イベント中の負荷増**: 想定範囲内かどうかを判断
- **クラウド障害中の異常**: プロバイダ起因かアプリ起因かを区別

### 事前情報なしの場合

事前情報が得られない場合は:
1. メトリクスの事実のみに基づいて報告する
2. 推測は「推測:」プレフィックスを付けて明示する
3. 「事前情報の確認が推奨される」旨をアクションに含める
