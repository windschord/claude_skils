# セルフレビューガイド

このガイドでは、サブエージェントへのプロンプト、差分取得、修正適用の手順を定義します。6観点の詳細なレビュー基準は[ai-code-reviewのレビューガイド](../../ai-code-review/references/review_guide_ja.md)を参照してください。

## サブエージェントプロンプト

### 共通ヘッダー（全エージェント共通）

以下をプロンプトの冒頭に含める:

```text
あなたはコードレビューエージェントです。指定された観点でローカルの変更差分をレビューし、結果をJSON形式のみで返してください。

## レビュー対象
- ベースブランチ: {base_branch}
- 変更ファイル一覧:
{file_list}

## 出力形式（厳守）
結果は以下のJSONのみを返してください。説明文・マークダウン・前置き・後書きは一切不要です。

{出力JSONスキーマ（SKILL.mdの「サブエージェントの出力フォーマット」セクションをそのまま貼付）}

## レビュールール
- 推測でレビューしない。必ずコードを読んで根拠を示す
- 数値・事実を根拠にする場合は必ずソースを確認する
- 変更範囲外のコードへの指摘は禁止（セキュリティ問題は例外）
- fix.old_text/fix.new_textは、Editツールにそのまま渡せる正確なテキストにする
- contextは該当行の前後3行程度を含める（行番号ズレがあっても位置特定できるように）
- 同一ファイル内に同じパターンの問題が複数ある場合でも、各箇所を個別のfindingとして報告する
- 指摘がない場合は findings: [] で返す
```

### エージェントA: セキュリティ + 既知脆弱性

```text
{共通ヘッダー}

## 担当観点
1. **セキュリティ** (category: security, iam-permissions)
2. **既知脆弱性の検出** (category: known-vulnerability)

## レビュー基準
ai-code-review/references/review_guide_ja.md の以下のセクションに従ってレビューしてください:
- 「観点1: セキュリティ」セクション全体
- 「観点6: 既知脆弱性の検出」セクション全体

## 特に注意すべき点
- OWASP Top 10（SQLインジェクション、XSS、コマンドインジェクション、パストラバーサル、SSRF）
- 認証・認可の欠如または不備
- 秘密情報（APIキー、パスワード、トークン）のハードコード
- 入力検証の欠如
- Terraformファイルの場合はIAM権限の過不足
- 依存パッケージの既知脆弱性（CVE）
- 暗号化アルゴリズムの選択（非推奨アルゴリズムの使用）

## 横断チェック
セキュリティ上の問題パターンを発見した場合、変更されたすべてのファイルで同じパターンがないか確認し、cross_checkに記録してください。
```

### エージェントB: 可読性・複雑度

```text
{共通ヘッダー}

## 担当観点
1. **可読性・複雑度** (category: readability)

## レビュー基準
ai-code-review/references/review_guide_ja.md の「観点3: 可読性・複雑度」セクションに従ってレビューしてください。

## 特に注意すべき点
- 命名規則（変数名・関数名・クラス名の意味の明確さ、略語の適切性、boolean変数のプレフィックス）
- 関数設計（単一責任の原則、引数の数、副作用、早期リターン）

## 横断チェック
```

```text
## 観点5のローカル用読み替えルール
PRではなくローカル変更のため、以下のように読み替えてください:
- 「PRタイトル」→ ブランチ名・最新コミットメッセージ
- 「PR説明文」→ コミットメッセージ全体（git log --oneline で確認）
- 「レビュアー向け情報」→ 将来のPR説明文に含めるべき情報の提案

## 特に注意すべき点
- README・API仕様書と実装の不一致
- コード内コメントと実際のロジックの乖離
- 同一ファイル内のステータス値の矛盾
- 新規ライブラリ追加時のライセンス・メンテナンス状況
- lockfileの更新漏れ
- コミットメッセージが変更内容を正確に反映しているか

## SDDプロジェクトの場合
docs/sdd/ ディレクトリが存在する場合、以下も確認してください:
- requirements/: 実装が要件を満たしているか
- design/: 実装が設計に沿っているか
- tasks/: タスクの受入基準を満たしているか

## 横断チェック
ドキュメント乖離のパターンを発見した場合、変更されたすべてのファイルで同じパターンがないか確認し、cross_checkに記録してください。
```

## 差分取得パターン

### ベースブランチの自動判定

```bash
BASE=$(git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's@^refs/remotes/origin/@@')
if [ -z "$BASE" ]; then
  git show-ref --verify --quiet refs/remotes/origin/main && BASE="main" || BASE="master"
fi
```

### 差分取得コマンド一覧

| ユースケース | コマンド |
|-------------|---------|
| ブランチ間の全変更（デフォルト） | `git diff ${BASE}...HEAD` |
| ステージ済みのみ | `git diff --cached` |
| 未コミットの全変更 | `git diff HEAD` |
| ワーキングツリーのみ | `git diff` |
| 特定ファイル | `git diff ${BASE}...HEAD -- path/to/file` |
| 特定ディレクトリ | `git diff ${BASE}...HEAD -- src/` |
| ファイル一覧のみ | `git diff ${BASE}...HEAD --name-only` |
| 統計情報 | `git diff ${BASE}...HEAD --stat` |

## 修正適用ガイド

### 修正の優先順位

1. critical → 必ず修正（ユーザー承認必須）
2. warning → 修正推奨（ユーザー承認必須）
3. suggestion → ユーザー判断
4. nitpick → ユーザー判断

### 修正適用手順

1. findingの`fix.old_text`/`fix.new_text`を取得
2. 対象ファイルをReadツールで読み込み（最新状態確認）
3. `old_text`がファイル内で一意であることを確認
4. Editツールで修正適用
5. 一意でない場合は`context`を使って周辺コードを確認し、正確な置換対象を特定

### 修正時の注意事項

- **1指摘ずつ適用**: 前の修正で後の行番号がズレる可能性がある
- **修正困難なケース**: `fix`で表現しきれない場合はユーザーに手順を提示
- **修正後確認**: critical/warning修正後はReadツールで再読み込みしリグレッション確認

## 再レビュー手順

### 前回レポートの読み込み

1. `.self-review/`から最新の`.json`ファイルを取得
2. `findings`配列から前回の指摘一覧を抽出
3. 各指摘のid・file・line・severity・fix情報を記録

### 修正状況の確認方法

各指摘について:
1. `git diff`で該当ファイルの変更を確認
2. `fix.old_text`が消えているか確認 → resolved候補
3. `fix.new_text`に相当する変更があるか確認 → resolved確定
4. 部分的にしか修正されていない → partially-resolved
5. 変更なし → unresolved
6. 修正により新たな問題発生 → regressed

### 再レビュー時のサブエージェントプロンプト追加情報

再レビュー時は、共通ヘッダーの後に以下を追加する:

```text
## 前回レビュー情報（再レビューモード）
前回のレビューで以下の指摘がありました。修正状況の確認は不要です（メインが行います）。
新規の変更のみをレビューしてください。ただし、前回指摘と関連する箇所に新たな問題があれば報告してください。

前回の指摘ID一覧:
{前回findingsのid, file, line, categoryのリスト}
```
