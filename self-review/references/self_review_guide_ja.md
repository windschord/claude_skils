# セルフレビューガイド

このガイドでは、セルフレビュースキル固有の手順を説明します。6観点の詳細なレビュー基準については[ai-code-reviewのレビューガイド](../../ai-code-review/references/review_guide_ja.md)を参照してください。

## 差分取得パターン

### ベースブランチの自動判定

```bash
# origin/HEAD からデフォルトブランチを取得
BASE=$(git symbolic-ref refs/remotes/origin/HEAD 2>/dev/null | sed 's@^refs/remotes/origin/@@')

# 取得できない場合のフォールバック
if [ -z "$BASE" ]; then
  # main が存在すれば main、なければ master
  if git show-ref --verify --quiet refs/remotes/origin/main; then
    BASE="main"
  else
    BASE="master"
  fi
fi
```

### 差分取得コマンド一覧

| ユースケース | コマンド | 説明 |
|-------------|---------|------|
| ブランチ間の全変更 | `git diff ${BASE}...HEAD` | デフォルト。PRに含まれる変更と同等 |
| ステージ済みのみ | `git diff --cached` | `git add` 済みの変更のみ |
| 未コミットの全変更 | `git diff HEAD` | ステージ済み + 未ステージ |
| ワーキングツリー | `git diff` | 未ステージの変更のみ |
| 特定ファイル | `git diff ${BASE}...HEAD -- path/to/file` | ファイル指定 |
| 特定ディレクトリ | `git diff ${BASE}...HEAD -- src/` | ディレクトリ指定 |
| ファイル一覧のみ | `git diff ${BASE}...HEAD --name-only` | 変更ファイルパスのみ |
| 統計情報 | `git diff ${BASE}...HEAD --stat` | ファイルごとの変更行数 |

### 差分の選択ガイド

```text
ユーザーの指定がない場合の優先順位:
1. ブランチ間差分（${BASE}...HEAD）を使用する
2. コミットされていない変更がある場合は警告を出す
3. ユーザーに差分対象を確認する
```

## 変更説明の適切性（観点5のローカル適用）

ai-code-reviewの「PR説明の適切性」をローカル変更に適用する際の読み替え:

| ai-code-review（PR） | self-review（ローカル） |
|---------------------|----------------------|
| PRタイトル | ブランチ名 / 最新コミットメッセージ |
| PR説明文 | コミットメッセージ全体（複数コミットの場合は`git log`） |
| レビュアー向け情報 | 将来のPR説明文に含めるべき情報の提案 |

### チェック項目

1. **コミットメッセージの適切性**
   - 変更内容を正確に反映しているか
   - Conventional Commits等の規約に従っているか（プロジェクトに規約がある場合）
   - 「なぜ」の情報が含まれているか

2. **変更の一貫性**
   - 関連しない変更が混在していないか
   - コミット粒度が適切か

3. **PR作成時の推奨事項**
   - PR説明文に含めるべき情報の提案（テスト手順、破壊的変更、マイグレーション等）

## 修正適用ガイド

### 修正の優先順位

```text
1. critical → 必ず修正（ユーザー承認必須）
2. warning → 修正推奨（ユーザー承認必須）
3. suggestion → ユーザー判断
4. nitpick → ユーザー判断
```

### 修正適用時の注意事項

1. **行番号のズレに注意**
   - 前の修正で行番号がズレる可能性がある
   - 各修正前にReadツールで最新のファイル内容を確認する
   - `old_text`の一意性を確保するため、十分なコンテキスト（前後の行）を含める

2. **修正の独立性を確認**
   - 複数の修正が同一ファイルの近接行にある場合、相互影響を確認する
   - 影響がある場合は修正順序を調整する

3. **修正後のビルド・テスト確認**
   - 修正適用後、可能であればビルドやテストを実行して確認する
   - テストが失敗した場合はユーザーに報告し、修正のロールバックを提案する

### 修正困難なケースの対応

old_text/new_textペアで表現しきれない修正（大規模なリファクタリング等）の場合:

```text
1. 修正方針と手順を具体的にユーザーに提示する
2. ユーザーの承認を得てから段階的に修正する
3. 各段階でビルド・テストの確認を挟む
```

## レポートファイル管理

### 出力先

```text
{プロジェクトルート}/.self-review/
└── review-YYYY-MM-DD-HHMMSS.md
```

### .gitignore への追加推奨

セルフレビューレポートはローカル用途のため、`.gitignore` に追加することを推奨する:

```text
# Self-review reports
.self-review/
```

### レポートの保持期間

- 最新3件のレポートを保持する
- 古いレポートは自動削除しない（ユーザーに削除を提案する）

## 再レビューの手順

### 前回レポートの読み込み

```text
1. .self-review/ ディレクトリから最新のレポートファイルを取得
2. レポート内のAI向けJSONから前回の指摘一覧を抽出
3. 各指摘のID・ファイル・行番号・重大度を記録
```

### 修正状況の確認

各指摘について以下を確認する:

| 確認項目 | 方法 |
|---------|------|
| コード変更 | `git diff` で指摘箇所の変更を確認 |
| 修正の妥当性 | 推奨修正と実際の修正を比較 |
| リグレッション | 修正周辺のコードに新たな問題がないか確認 |

### 判定基準

| 状態 | 判定条件 |
|------|---------|
| resolved | 指摘どおりまたは同等以上の品質で修正済み |
| partially-resolved | 一部修正されたが不十分な箇所が残る |
| unresolved | 未修正（コードに変更なし） |
| wont-fix | ユーザーが修正しない判断をした（レポートにコメント記載） |
| regressed | 修正により別の問題が発生 |

## SDDワークフローとの連携

### 推奨実行タイミング

```text
SDDワークフローでの推奨フロー:

1. task-executing: タスク実装
2. task-executing: 逆順レビュー（ドキュメント整合性）
     tasks/ → design/ → requirements/
3. self-review: コード品質レビュー（6観点）
     → critical/warningがあれば修正
4. task-executing: タスクDONEマーク + コミット
5. PR作成
```

### SDDドキュメントとの照合

SDDプロジェクトでは「ドキュメントとの乖離」観点で以下も確認する:

| 照合対象 | 確認内容 |
|---------|---------|
| docs/sdd/requirements/ | 実装が要件を満たしているか |
| docs/sdd/design/ | 実装が設計に沿っているか |
| docs/sdd/tasks/ | タスクの受入基準を満たしているか |

これは逆順レビューと重複する部分があるため、逆順レビュー実施済みの場合は結果を参照し、重複指摘を避ける。

## エージェントチームレビューの手順

### チームリーダーの動作

```text
1. 変更差分を取得しファイル一覧を把握
2. ファイルを観点別に分配
3. Agent toolで各レビュアーを起動（subagent_type未指定 = general-purpose）
4. 各レビュアーの結果を統合
5. 重複する指摘を統合・整理
6. 統合結果をユーザーに報告
```

### レビュアーへの指示テンプレート

```text
以下のファイルの変更差分を[担当観点]の観点でレビューしてください。

対象ファイル:
- [ファイルパス一覧]

ベースブランチ: [BASE]

レビュー基準は ai-code-review/references/review_guide_ja.md の[担当観点]セクションに従ってください。

指摘は以下のJSON形式で返してください:
{
  "findings": [
    {
      "id": "F-NNN",
      "file": "ファイルパス",
      "line": "行番号",
      "severity": "critical|warning|suggestion|nitpick",
      "category": "カテゴリ",
      "title": "問題の端的な説明",
      "description": "詳細な説明",
      "fix": {
        "old_text": "置換対象",
        "new_text": "置換後"
      },
      "fix_strategy": {
        "approach": "修正アプローチ",
        "alternatives": [],
        "impact": "影響範囲",
        "effort": "small|medium|large"
      }
    }
  ]
}
```
