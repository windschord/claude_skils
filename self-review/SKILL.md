# セルフレビュースキル

ローカルの変更差分を体系的にレビューし、PR作成前に品質問題を検出・修正するスキルです。ai-code-reviewと同一の6観点・重大度基準を使用しますが、GitHub APIに依存せずローカルで完結します。指摘事項は直接修正（Editツール）またはレポート出力のいずれかで対応します。

## 重要な原則

```text
+---------------------------------------------------------------+
| セルフレビューの鉄則                                             |
+---------------------------------------------------------------+
| 1. 推測でレビューしない - 必ずコードを読んで根拠を示す              |
| 2. 6つの観点を漏れなくチェックする                                |
| 3. 指摘には必ず根拠（該当コード・行番号）を添える                   |
| 4. 数値・事実を根拠にする場合は必ずソースを確認する                  |
| 5. 重大度を正確に分類し、過剰な指摘を避ける                        |
| 6. critical/warningはユーザー承認なしに修正しない                  |
| 7. suggestion/nitpickも修正前にユーザーに提示する                 |
+---------------------------------------------------------------+
```

## ai-code-reviewとの違い

| 項目 | ai-code-review | self-review |
|------|---------------|-------------|
| 対象 | GitHub PR（リモート） | ローカルの変更差分（git diff） |
| 差分取得 | `gh pr diff` | `git diff`（ステージ済み/未ステージ/ブランチ間） |
| 出力先 | PRコメント（GitHub API） | コンソール出力 + オプションでレポートファイル |
| 修正 | 指摘のみ（修正はpr-comment-fixerが担当） | 指摘 + 直接修正（ユーザー承認後にEditツール適用） |
| 再レビュー | PR上の前回コメントを参照 | 前回レポートファイルを参照 |
| PR説明の観点 | PRタイトル・説明文を評価 | コミットメッセージ・変更の意図を評価 |
| 使用タイミング | PR作成後 | PR作成前・タスク完了前 |

## レビュー観点（6つ）

| # | 観点 | 概要 |
|---|------|------|
| 1 | セキュリティ | OWASP Top 10、認証・認可、入力検証、秘密情報の漏洩、暗号化、Terraform IAM権限の過不足チェック |
| 2 | ドキュメントとの乖離 | README、API仕様書、設計書、コメントと実装の不一致 |
| 3 | 可読性・複雑度 | 命名、関数の長さ、ネスト深度、循環的複雑度、コード重複、SOLID原則 |
| 4 | ライブラリ選定の妥当性 | ライセンス、メンテナンス状況、セキュリティ履歴、サイズ、代替手段 |
| 5 | 変更説明の適切性 | コミットメッセージ・変更の意図が変更内容を正確に反映しているか |
| 6 | 既知脆弱性の検出 | 使用ライブラリのバージョンにCVE等の既知脆弱性が報告されていないか |

### カテゴリの細分類

ai-code-reviewと同一のカテゴリ体系を使用する。

| 観点 | category値 | 使い分け |
|------|-----------|---------|
| セキュリティ | `security` | セキュリティ全般 |
| セキュリティ | `iam-permissions` | Terraform IAM権限の過不足 |
| ドキュメント乖離 | `docs-drift` | 異なるドキュメント間の不一致 |
| ドキュメント乖離 | `internal-consistency` | 同一ファイル内の論理矛盾 |
| 可読性・複雑度 | `readability` | 可読性・複雑度全般 |
| ライブラリ選定 | `library` | ライブラリ選定全般 |
| 変更説明 | `change-description` | コミットメッセージ・変更意図と実際の変更の整合性 |
| 既知脆弱性 | `known-vulnerability` | CVE・セキュリティアドバイザリ |

## このスキルを使用する場面

### PR作成前のセルフレビュー

- 実装完了後、PRを作成する前に品質を確認したい場合
- コードの問題を早期に発見し、レビュー指摘を未然に防ぎたい場合

### SDDタスク完了前のチェック

- task-executingの逆順レビューと併用し、コード品質も検証する場合
- タスクをDONEにする前の最終チェックとして使用する場合

### 特定ファイルの変更レビュー

- 特定のファイルやディレクトリの変更のみをレビューしたい場合

## ワークフロー概要

```text
レビュー対象の指定（ブランチ比較 / ステージ済み / 全変更 / ファイル指定）
      |
1. 変更差分の取得（git diff）
      |
2. 変更ファイルの分類と優先度付け
      |
3. 6観点でのレビュー実行
   a. セキュリティ
   b. ドキュメントとの乖離
   c. 可読性・複雑度
   d. ライブラリ選定の妥当性
   e. 変更説明の適切性
   f. 既知脆弱性の検出
      |
3.5. 横断チェック（同一パターンの問題を他ファイルでも確認）
      |
4. 指摘の重大度分類と整理
      |
5. ★ レビュー結果の報告 ★
   - 指摘件数・重大度の内訳を出力
   - 各指摘の詳細を出力
      |
6. ★ 修正方針の承認確認 ★（critical/warningがある場合は必須ゲート）
   - AskUserQuestionで修正方針を確認
      |
7. 承認された修正の適用（Editツール）
      |
8. 修正後の再チェック（必要な場合）
```

## 各ステップの詳細

### ステップ1: 変更差分の取得

ユーザーの指定に応じて適切な差分を取得する。

```bash
# デフォルト: メインブランチとの差分
git diff main...HEAD

# ステージ済み変更のみ
git diff --cached

# 未コミットの全変更
git diff HEAD

# 特定ファイル/ディレクトリのみ
git diff main...HEAD -- src/

# 変更ファイル一覧
git diff main...HEAD --name-only

# ベースブランチの自動判定
git symbolic-ref refs/remotes/origin/HEAD | sed 's@^refs/remotes/origin/@@'
```

ベースブランチの判定:
1. ユーザーが明示的に指定した場合はそれを使用
2. 指定がない場合は`origin/HEAD`（通常は`main`または`master`）を使用
3. SDDプロジェクトの場合はタスクのブランチ情報を参照

### ステップ2: 変更ファイルの分類

ai-code-reviewと同一の優先度基準を使用する。

| 優先度 | カテゴリ | 例 |
|--------|---------|-----|
| 高 | セキュリティ関連 | 認証、認可、暗号化、入力処理、Terraform IAMポリシー |
| 高 | API / データベース | エンドポイント、マイグレーション、スキーマ |
| 中 | ビジネスロジック | ドメインロジック、サービス層 |
| 中 | 設定・インフラ | CI/CD、Docker、依存関係 |
| 低 | UI/表示 | テンプレート、スタイルシート |
| 低 | テスト | テストコード（ただしセキュリティテストは優先度高） |
| 低 | ドキュメント | README、コメント更新 |

### ステップ3: 6観点でのレビュー実行

各観点で変更コードを精査する。指摘はすべて以下の情報を含むこと:

| 項目 | 内容 |
|------|------|
| ファイル | 対象ファイルパス |
| 行番号 | 該当行（範囲も可） |
| 重大度 | critical / warning / suggestion / nitpick |
| 観点 | カテゴリ細分類を参照 |
| 指摘内容 | 何が問題で、なぜ問題なのか |
| 推奨修正 | どう直すべきか（old_text / new_textペア） |

**重大度の定義:**

| 重大度 | 意味 | 修正要否 |
|--------|------|---------|
| critical | セキュリティ脆弱性、データ損失の可能性、本番障害のリスク | 修正必須 |
| warning | バグの可能性、パフォーマンス問題、設計上の懸念 | 修正推奨 |
| suggestion | より良い実装方法の提案、リファクタリング案 | 任意 |
| nitpick | スタイル、命名、些細な改善 | 任意 |

**各観点の詳細**: [../ai-code-review/references/review_guide_ja.md](../ai-code-review/references/review_guide_ja.md)（観点5のみ本スキル用にローカル変更向けに読み替えること）

### ステップ3.5: 横断チェック

```text
指摘を発見した場合の横断チェック手順:
1. 指摘のパターンを抽出する
2. 変更されたすべてのファイルで同様のパターンを検索する
3. 変更されていないが関連するファイルも確認対象に含める
4. 同一パターンの問題が複数見つかった場合はまとめて指摘する
```

### ステップ4: 指摘の整理

- 重大度の高い順にソート
- 同一ファイル内の指摘はまとめる
- 重複する指摘は統合
- 過度に些細な指摘（nitpick）が多い場合は厳選する

### ステップ5: レビュー結果の報告

レビュー結果をユーザーに出力する。

```text
## セルフレビュー結果

| 重大度 | 件数 |
|--------|------|
| critical | X件 |
| warning | X件 |
| suggestion | X件 |
| nitpick | X件 |

### 指摘一覧

1. **[critical]** `src/auth.ts:42` - セキュリティ: SQLインジェクションの可能性
2. **[warning]** `src/utils.ts:15` - 可読性: 関数の循環的複雑度が高い（12）
...
```

### ステップ6: 修正方針の承認確認

critical または warning の指摘がある場合、修正前にユーザーの承認を得ること。

```text
AskUserQuestion:
  question: "上記の指摘について、どのように対応しますか？"
  options:
    - label: "すべて修正する"
      description: "critical/warningの指摘をすべて修正します"
    - label: "選択して修正する"
      description: "修正する指摘を個別に選択します"
    - label: "レポートのみ出力"
      description: "修正は行わず、レビュー結果をレポートファイルに出力します"
    - label: "対応しない"
      description: "レビュー結果を確認のみとし、修正は行いません"
```

### ステップ7: 修正の適用

承認された指摘について、Editツールを使って修正を適用する。

```text
修正適用の手順:
1. 指摘のold_text/new_textペアを確認
2. 対象ファイルをReadツールで読み込み（最新状態の確認）
3. Editツールで修正を適用
4. 適用結果を報告
```

修正適用時のルール:
- **1指摘ずつ適用**: 一括適用せず、指摘ごとにEditツールで修正する
- **修正競合の確認**: 前の修正が後の修正に影響する場合は行番号を再確認する
- **修正困難な場合**: old_text/new_textで表現しきれない修正は、手順を具体的に提示してユーザーに判断を委ねる

### ステップ8: 修正後の再チェック

critical/warningの修正を適用した場合、修正箇所を再度確認する。

```text
再チェックの手順:
1. 修正されたファイルを再読み込み
2. 修正が意図通りか確認（リグレッションチェック）
3. 修正により新たな問題が発生していないか確認
4. 結果を報告
```

## レポートファイル出力（オプション）

ユーザーが「レポートのみ出力」を選択した場合、レビュー結果をファイルに出力する。

出力先: プロジェクトルートの `.self-review/` ディレクトリ

```text
.self-review/
└── review-YYYY-MM-DD-HHMMSS.md
```

**レポートテンプレート**: [assets/templates/self_review_report_template_ja.md](assets/templates/self_review_report_template_ja.md)

## 再レビュー（前回レポートとの差分確認）

前回のセルフレビューレポートが存在する場合、再レビューとして前回指摘の修正状況を確認できる。

### 手順

1. `.self-review/` から最新のレポートファイルを読み込む
2. 前回指摘の各項目について修正状況を確認:

| 状態 | 説明 |
|------|------|
| resolved | 修正済み |
| partially-resolved | 一部修正されたが不十分 |
| unresolved | 未修正 |
| wont-fix | 修正しない判断がされた |
| regressed | 修正により別の問題が発生 |

3. 新規変更に対する6観点レビューを実施
4. 結果を統合して報告

## エージェントチームによる並列レビュー（条件を満たす場合は使用）

### 適用条件

1. 変更ファイル数が10以上
2. 変更が3つ以上の独立したモジュール/ディレクトリにまたがる
3. 各モジュールの変更が独立してレビュー可能

### チーム構成

ai-code-reviewと同一のチーム構成を使用する。

```text
チームリーダー（レビュー統合・結果報告に専念）
    |
    +-- レビュアーA: セキュリティ・既知脆弱性観点担当
    +-- レビュアーB: 可読性・複雑度観点担当
    +-- レビュアーC: ドキュメント乖離・ライブラリ・変更説明観点担当
```

## SDDワークフローとの連携

### task-executingとの併用

SDDワークフローでは、task-executingの逆順レビュー（ドキュメント整合性）と本スキル（コード品質）を組み合わせて使用する。

```text
タスク実装完了
    |
    +-- 逆順レビュー（task-executing）: ドキュメント整合性
    |     tasks/ → design/ → requirements/
    |
    +-- セルフレビュー（self-review）: コード品質
          6観点のレビュー → 修正 → PR作成
```

### 実行タイミング

- **推奨**: タスクのDONEマーク前・PR作成前
- **任意**: 実装中の中間チェック

## 禁止事項

```text
- 推測に基づくレビュー（コードを読まずに指摘する）
- 個人の好みだけを根拠にした指摘
- 裏取りなしに数値・事実を断定する指摘
- 変更範囲外のコードへの指摘（ただしセキュリティ問題は例外）
- 過剰な nitpick
- ユーザーの承認なしにcritical/warningの修正を適用すること
- レビュー結果を報告せず修正だけ行うこと（必ず先に結果を提示する）
```
