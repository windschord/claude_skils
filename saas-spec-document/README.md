# SaaS Service Specification Document Skill

SaaSサービス向けのサービス仕様書を作成するスキルです。運用設計書や要件定義書をインプットとして活用し、経済産業省「SaaS向けSLAガイドライン」に準拠した包括的なサービス仕様書を生成します。

## 主な機能

- **インプットドキュメント活用**: 運用設計書・要件定義書からの情報抽出
- **ガイドライン準拠**: 経済産業省「SaaS向けSLAガイドライン」に準拠
- **7カテゴリの網羅**: 可用性、信頼性、データ管理、セキュリティ、サポート、拡張性、コンプライアンス
- **段階的作成**: 各セクションでユーザー確認を取りながら作成
- **一貫性チェック**: インプットドキュメントとの整合性を自動検証

## 使用開始

```
サービス仕様書を作成したい
```

または

```
SaaSのSLA文書を作成してほしい
```

## ワークフロー

```
1. インプットドキュメントの確認
   - 運用設計書の読み込み
   - 要件定義書の読み込み
   ↓
2. 基本情報のヒアリング
   - サービス概要
   - 対象顧客
   - 料金プラン
   ↓
3. カテゴリ別情報の収集
   - インプットからの抽出
   - 不足情報のヒアリング
   ↓
4. サービス仕様書の段階的作成
   - 各セクション作成
   - ユーザー確認・承認
   ↓
5. 一貫性チェック
   ↓
6. 最終レビューと承認
```

## インプットドキュメントの活用

### 運用設計書からの抽出

| 運用設計書のセクション | サービス仕様書への反映先 |
|----------------------|------------------------|
| SLO/SLI定義 | 信頼性 - サービスレベル目標 |
| SLA定義 | 可用性 - サービス稼働率 |
| 監視項目 | 信頼性 - サービスレベル指標 |
| インシデント管理 | 信頼性 - 障害時の対応時間 |
| バックアップ設計 | データ管理 - バックアップ |
| DR設計 | 可用性 - ディザスタリカバリ |
| セキュリティ運用 | セキュリティ |
| 運用体制 | サポート - サポート体制 |

### 要件定義書からの抽出

| 要件定義書のセクション | サービス仕様書への反映先 |
|----------------------|------------------------|
| 非機能要件（可用性） | 可用性 |
| 非機能要件（性能） | 信頼性 - SLO |
| 非機能要件（セキュリティ） | セキュリティ |
| 非機能要件（拡張性） | 拡張性 |
| ユーザーストーリー | サービス概要 - ユースケース |
| 機能要件 | サービス概要 - サービスの機能 |

## サービス仕様書の構成

経済産業省「SaaS向けSLAガイドライン」に基づく7カテゴリ：

### 1. サービス概要
- サービス基本情報
- サービスの目的と価値
- 機能一覧
- 提供形態

### 2. 可用性（Availability）
- サービス時間
- サービス稼働率（SLA）
- 計画停止予定通知
- サービス提供終了時の事前通知
- ディザスタリカバリ
- アップグレード方針

### 3. 信頼性（Reliability）
- サービスレベル目標（SLO）
- サービスレベル指標（SLI）
- エラーバジェット
- 障害時の対応時間

### 4. データ管理（Data Management）
- データの取り扱い
- バックアップ
- データ復旧目標（RTO/RPO）
- データ暗号化
- 解約時のデータ取り扱い

### 5. セキュリティ（Security）
- セキュリティ認証・認定
- 通信セキュリティ
- 認証・認可
- マルチテナントセキュリティ
- 脆弱性管理

### 6. サポート（Support）
- サポート体制
- サポートチャネル
- サポート対応レベル
- セルフサービス

### 7. 拡張性（Scalability）
- カスタマイズ性
- 外部接続性（API）
- API制限
- リソース上限

### 8. コンプライアンス（Compliance）
- 法令遵守
- 監査対応
- 監査ログ
- データ所在地

### 9. 料金・課金
- 料金プラン
- 課金方式
- SLA未達時の補償

### 10. 責任分界
- 責任共有モデル
- 責任範囲の詳細
- 免責事項

## 使用例

### 運用設計書からサービス仕様書を作成

```
ユーザー: 運用設計書を基にサービス仕様書を作成したい

コンサルタント: 運用設計書を確認します。ファイルパスを教えてください。

ユーザー: docs/operations_design.md

コンサルタント: 運用設計書を確認しました。以下の情報を抽出できます：
                ✅ SLO定義: 可用性99.9%、レスポンスタイム500ms以下
                ✅ バックアップ: 日次フル、6時間毎差分、30日保持
                ✅ インシデント対応時間: P1=15分以内...

                これらをサービス仕様書に反映してよろしいですか？
```

### ゼロから作成

```
ユーザー: 新しいSaaSサービスのサービス仕様書を作成したい

コンサルタント: サービス仕様書を作成します。
                まず、既存のドキュメントはお持ちですか？
                - 運用設計書
                - 要件定義書
                - その他の仕様書

ユーザー: まだドキュメントはない

コンサルタント: では基本情報からヒアリングを開始します。
                Q1: サービス名を教えてください
                Q2: サービスの提供形態は？
                A) マルチテナント型
                B) シングルテナント型
                C) ハイブリッド型
```
