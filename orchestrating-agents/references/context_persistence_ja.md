# コンテキスト永続化

## 概要

親・子セッションがコンテキスト圧縮されても状態を維持するための永続化メカニズム。`.orchestrating-agents/` ディレクトリに全ての状態をファイルとして保存し、圧縮後に再読み込みすることで作業を継続する。

---

## .orchestrating-agents/ ディレクトリ構成

```text
.orchestrating-agents/
├── mission.md                  # 親のミッション定義
├── task_ledger.md              # タスク台帳（親のみ更新、ユーザー確認用）
├── session_state.md            # セッション状態（レジューム用）
└── workorders/                 # 子のワークオーダー
    ├── TASK-001.md
    ├── TASK-002.md
    └── TASK-003.md
```

---

## .gitignore への追加

親がミッションを定義する際、`.orchestrating-agents/` が `.gitignore` に登録されていなければ追加する。

```text
# .gitignore に以下を追加
.orchestrating-agents/
```

**理由:** オーケストレーション状態は一時的なセッション情報であり、リポジトリにコミットすべきではない。

---

## mission.md の構造と更新タイミング

### 構造

```markdown
# Mission

## 原文指示
<ユーザーの原文指示をそのまま保存。編集・要約しない>

## 目標
<解釈した達成目標>

## 成功基準
- <基準1>
- <基準2>
- <基準3>

## 実行計画
### フェーズ1: <フェーズ名>
- タスク: TASK-001, TASK-002
- 並列実行: 可
- 見込み時間: <目安>

### フェーズ2: <フェーズ名>
- タスク: TASK-003
- 依存: フェーズ1の完了
- 見込み時間: <目安>

## 現在のキュー状態
- 進行中: TASK-001（child-task-exec）
- 次: TASK-002
- 残り: TASK-003, TASK-004
```

### 更新タイミング

| イベント | 更新内容 |
|---------|---------|
| ミッション定義時 | 全セクションを作成 |
| タスク開始/完了時 | 「現在のキュー状態」を更新 |
| 軌道変更時 | 「実行計画」と「現在のキュー状態」を更新 |
| ユーザー追加指示時 | 「原文指示」に追記、「実行計画」を更新 |

### 更新者

**親のみ**が mission.md を更新する。子・孫は読み取り専用。

## task_ledger.md の構造と更新タイミング

### 構造

```markdown
# Task Ledger

## 進行中
| ID | タスク | ステータス | 担当 | 開始時刻 |
|----|--------|-----------|------|---------|
| TASK-001 | API実装 | IN_PROGRESS | child-task-exec | 2026-03-15T10:00 |

## キュー（待機中）
| ID | タスク | 依存先 | 優先度 |
|----|--------|--------|--------|
| TASK-002 | DB実装 | - | 高 |
| TASK-003 | 結合テスト | TASK-001, TASK-002 | 中 |

## 完了（直近10件）
| ID | タスク | 完了時刻 | 成果物 |
|----|--------|---------|--------|
| TASK-000 | 要件定義 | 2026-03-15T09:30 | docs/sdd/requirements/ |

## アーカイブ
10件を超えた完了タスクは以下に移動:
→ .orchestrating-agents/archive/completed_tasks.md

## 軌道変更履歴
### CC-001: <変更概要>
- 日時: <タイムスタンプ>
- 対象タスク: <ID>
- 変更内容: <詳細>
```

### 完了タスクの管理

- 完了タスクは直近10件のみ task_ledger.md に保持する
- 10件を超えた古い完了タスクは `.orchestrating-agents/archive/completed_tasks.md` に移動する
- アーカイブファイルには全完了タスクの履歴を保持する

### 更新タイミング

| イベント | 更新内容 |
|---------|---------|
| タスク登録時 | 「キュー」セクションに追加 |
| タスク開始時 | 「キュー」から「進行中」に移動 |
| タスク完了時 | 「進行中」から「完了」に移動 |
| タスクブロック時 | 「進行中」のステータスを BLOCKED に変更 |
| 軌道変更時 | 「軌道変更履歴」に追記 |
| 完了タスク超過時 | 古い完了タスクをアーカイブに移動 |

### 更新者

**親のみ**が task_ledger.md を更新する。子・孫は読み取り専用。

## workorders/TASK-XXX.md の構造と更新タイミング

### 構造

```markdown
# Workorder: TASK-XXX

## 親からの指示内容
<親から受け取った指示の全文をそのまま保存>

## タスク概要
- タスクID: TASK-XXX
- タスク種別: <種別>
- 目的: <目的>
- 期待成果物: <成果物>

## 作業分解（孫への作業単位）
### W01: <作業名>
- 担当: worker-<name>
- ステータス: DONE
- 対象ファイル: <パス一覧>
- 成果物: <成果物パス>

### W02: <作業名>
- 担当: worker-<name>
- ステータス: IN_PROGRESS
- 対象ファイル: <パス一覧>
- 成果物: <成果物パス>

### W03: <作業名>
- 担当: -（未着手）
- ステータス: TODO
- 対象ファイル: <パス一覧>
- 成果物: <成果物パス>

## 各孫の進捗状態
| 作業ID | 孫名 | ステータス | 備考 |
|--------|------|-----------|------|
| W01 | worker-impl-api | DONE | 正常完了 |
| W02 | worker-impl-ui | IN_PROGRESS | - |
| W03 | - | TODO | W01完了待ち |

## 軌道変更履歴
### CC-001: <変更概要>
- 受信日時: <タイムスタンプ>
- 変更内容: <詳細>
- 対応: <どのように反映したか>
- 影響する作業: W02, W03
```

### 更新タイミング

| イベント | 更新内容 |
|---------|---------|
| 子の起動時 | ファイルを作成、親の指示内容を記録 |
| 作業分解時 | 「作業分解」セクションを作成 |
| 孫の起動時 | 該当作業のステータスを IN_PROGRESS に更新、担当名を記録 |
| 孫の完了時 | 該当作業のステータスを DONE に更新、成果物を記録 |
| 軌道変更受信時 | 「軌道変更履歴」に追記、影響する作業を更新 |

### 更新者

**子が作成・更新**する。孫は読み取り専用。マージ時にメインブランチに反映される。

## Worktree との関係

### 親のワークスペース（メインブランチ）

- `mission.md` -- 親のみが更新
- `task_ledger.md` -- 親のみが更新
- `session_state.md` -- 親のみが更新

### 子の Worktree（分岐ブランチ）

- `workorders/TASK-XXX.md` -- 子が作成・更新
- マージ時にメインブランチの `.orchestrating-agents/workorders/` に反映

### 更新の流れ

```text
1. 子が worktree 上で workorders/TASK-XXX.md を作成・更新
2. 子がコミット
3. 子の完了後、親が子の worktree ブランチをメインブランチにマージ
4. マージにより workorders/TASK-XXX.md がメインに反映
5. 親が task_ledger.md を更新
```

### 注意事項

- `mission.md` と `task_ledger.md` は親の worktree（メインブランチ）上にのみ存在
- 子は自身の worktree から `mission.md` を読み取ることはできるが、更新してはならない
- 子が `task_ledger.md` の内容を知りたい場合は、親に報告を要求する

## コンテキスト圧縮後の再読み込み手順

### 親セッションが圧縮された場合

```text
1. 親: .orchestrating-agents/mission.md を読み込み
   → ユーザーの原文指示、目標、実行計画を復元
2. 親: .orchestrating-agents/task_ledger.md を読み込み
   → 進行中/キュー/完了のタスク状態を復元
3. 親: .orchestrating-agents/session_state.md を読み込み
   → 子セッションの状態（name, agent_id）を復元
4. 親: 進行中の子セッションが存在すれば、完了を待機
5. 親: キューに pending タスクがあれば、次のタスクを実行
```

### 子セッションが圧縮された場合

```text
1. 子: .orchestrating-agents/workorders/TASK-XXX.md を読み込み
   → 親の指示内容、作業分解、孫の進捗状態を復元
2. 子: 進行中の孫セッションが存在すれば、完了を待機
3. 子: 未完了の作業があれば、新しい孫を起動して続行
4. 子: 全作業完了後、親に報告
```

### 再読み込み時の整合性チェック

```text
1. task_ledger.md のステータスと TaskList の状態が一致するか確認
2. 不一致がある場合は task_ledger.md を正とする
3. workorder のステータスと実際のファイル状態を照合
4. 成果物ファイルが存在するか確認
```
