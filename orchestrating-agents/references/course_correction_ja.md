# 軌道変更メカニズム

## 概要

ユーザーが進行中のタスクに対して方針変更を指示した場合、親（Director）は Agent tool の `name` パラメータと `SendMessage` を使って、進行中の子（Manager）にリアルタイムで変更を伝達する。本ドキュメントでは、軌道変更の具体的な手順、記録方法、影響範囲の評価、子から孫への変更伝搬について詳述する。

---

## 軌道変更の基本フロー

### 前提: 子セッションに名前を付けて起動

子セッションは必ず `name` パラメータを指定して起動する。この名前が SendMessage の宛先となる。

```text
Agent tool(
  subagent_type: "general-purpose",
  name: "child-task-exec",
  run_in_background: true,
  prompt: <指示内容>
)
```

### 軌道変更の手順

```text
1. 親: Agent tool(name: "child-task-exec", run_in_background: true) で子を起動（起動済み）
2. ユーザー: 方針変更を親に伝える
   例: 「APIのレスポンス形式をJSONからProtobufに変更して」
3. 親: 変更内容を解釈し、影響範囲を評価
4. 親: SendMessage(to: "child-task-exec", message: 方針変更内容) で子に伝達
5. 子: 受信した変更を反映して作業を調整
6. 親: task_ledger.md の軌道変更履歴に記録
7. 親: mission.md の実行計画を更新（必要な場合）
```

### SendMessage の具体的な使い方

```text
SendMessage(
  to: "child-task-exec",
  message: "## Course Correction\n\n### 変更内容\nAPIレスポンス形式をJSONからProtobufに変更\n\n### 理由\nユーザーからの方針変更指示\n\n### 影響範囲\n- api/handlers/ 配下のレスポンス生成処理\n- api/schemas/ 配下の型定義\n- tests/api/ 配下のテスト\n\n### 対応方針\n- 進行中のJSON実装を中断\n- Protobuf定義ファイルを先に作成\n- ハンドラーをProtobuf出力に変更"
)
```

## task_ledger.md の軌道変更履歴への記録

### 記録フォーマット

task_ledger.md の「軌道変更履歴」セクションに以下を記録する:

```markdown
## 軌道変更履歴

### CC-001: APIレスポンス形式の変更
- 日時: 2026-03-15T14:30:00
- 対象タスク: TASK-003（API実装）
- 変更前: JSONレスポンス
- 変更後: Protobufレスポンス
- 理由: ユーザーからの方針変更指示
- 影響タスク: TASK-003, TASK-005（結合テスト）
- ステータス: 伝達済み
```

### 記録タイミング

- 軌道変更を子に伝達した直後
- 子が変更の反映完了を報告した際にステータスを更新

## 変更の影響範囲評価

### 評価フロー

親が軌道変更を子に伝達する前に、以下の評価を行う:

```text
1. 変更が影響するタスクを特定
   - 進行中のタスク（直接影響）
   - キュー内の未実行タスク（間接影響）
   - 完了済みタスク（手戻りの可能性）

2. 影響レベルを判定
   - 軽微: 進行中タスク内の局所的変更のみ
   - 中程度: 複数タスクへの影響あるが手戻りなし
   - 重大: 完了済みタスクの手戻りが発生

3. 影響レベルに応じた対応
   - 軽微: 子に伝達して続行
   - 中程度: 影響タスク一覧をユーザーに提示し、確認後に伝達
   - 重大: 手戻りの工数をユーザーに提示し、承認を得てから伝達
```

### 影響レベルの判定基準

| レベル | 条件 | 対応 |
|--------|------|------|
| 軽微 | 変更が1タスク内に閉じる | 子に直接伝達 |
| 中程度 | 変更が未実行タスクにも影響する | ユーザーに影響範囲を提示後、伝達 |
| 重大 | 完了済みタスクの手戻りが発生する | ユーザーに工数を提示し、承認後に伝達 |

## 進行中タスクの中断・修正判断基準

### 中断すべきケース

- 変更により現在の作業が無駄になる場合
- 変更の前提が確定しないと続行できない場合
- 変更が対象ファイルに直接影響する場合

### 続行して後から修正すべきケース

- 変更が現在の作業と独立している場合
- 現在のステップを完了してから修正した方が効率的な場合
- 変更が軽微でインクリメンタルに適用可能な場合

### 判断フロー

```text
子が軌道変更を受信
  │
  ├─ 現在の作業が変更で無駄になるか?
  │   ├─ YES → 即座に中断、新方針で再開
  │   └─ NO ─┐
  │           │
  ├─ 現在のステップ完了まで続行可能か?
  │   ├─ YES → 現ステップ完了後に変更を適用
  │   └─ NO → 即座に中断、新方針で再開
```

## 子から孫への変更伝搬

### 孫が稼働中の場合

子が軌道変更を受信し、稼働中の孫に影響がある場合:

```text
1. 子が影響を受ける孫を特定
2. 子が SendMessage で孫に変更を伝達
   SendMessage(
     to: "worker-impl-api",
     message: "## Course Correction\n\n<変更内容>"
   )
3. 孫が変更を受信し、作業を調整
4. 影響を受けない孫はそのまま続行
```

### 孫がまだ起動していない場合

子が軌道変更を受信し、まだ起動していない孫の作業に影響がある場合:

```text
1. 子が workorder の該当作業を更新
2. 孫の起動時に更新後の指示を渡す
```

### 孫が既に完了している場合

子が軌道変更を受信し、完了済みの孫の成果物に影響がある場合:

```text
1. 子が手戻りの必要性を評価
2. 手戻りが必要な場合:
   a. 新しい孫を起動して修正作業を実行
   b. または子が自ら修正（軽微な場合）
3. 手戻り不要の場合: そのまま続行
```

---

## 軌道変更が連鎖する場合

### 複数タスクへの波及

```text
1. 親がキュー内の全タスクを走査
2. 影響を受けるタスクを特定
3. 影響の種類に応じて対応:
   - 未実行タスク: タスクの指示内容を更新
   - 進行中タスク: 子に SendMessage で伝達
   - 完了済みタスク: 手戻りタスクをキューに追加
4. task_ledger.md に全ての影響を記録
```

### 軌道変更の競合

短時間に複数の軌道変更が発生した場合:

```text
1. 後の変更が前の変更を上書きする場合: 最新の変更のみ適用
2. 変更が独立している場合: 両方の変更を適用
3. 変更が矛盾する場合: ユーザーに確認してから伝達
```
