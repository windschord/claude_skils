# エスカレーション方針

## 概要

3階層エージェント構造では、全ての判断をユーザーに確認していては効率が落ちる。一方で、重要な判断を勝手に行うとリスクがある。本ドキュメントでは、子（Manager）が代理承認できる判断と、親（Director）を経由してユーザーにエスカレーションすべき判断を明確に定義する。

---

## 子が代理承認できる判断レベル

以下の判断は子が自律的に行い、事後報告として親に含める。ユーザーへの確認は不要。

### 技術的トレードオフ

| 判断内容 | 例 | 条件 |
|---------|-----|------|
| ライブラリ選択 | 同等機能の2つのライブラリからの選択 | 既存プロジェクトの技術スタックと整合すること |
| 軽微なリファクタリング | 関数の分割、変数名の改善 | 外部インターフェースを変更しないこと |
| コードスタイルの判断 | 実装パターンの選択 | プロジェクトの既存スタイルに従うこと |
| テスト戦略の微調整 | テストケースの追加・修正 | カバレッジを下げないこと |

### 軽微な修正

| 判断内容 | 例 | 条件 |
|---------|-----|------|
| typo修正 | コメント・ドキュメントの誤字修正 | ロジックに影響しないこと |
| インポート追加 | 不足しているインポート文の追加 | 新しい依存関係を追加しないこと |
| フォーマット修正 | インデント、空白の調整 | リンターの指摘に従った修正 |
| ログ追加 | デバッグ・トレースログの追加 | パフォーマンスに影響しないこと |

### リトライ判断

| 判断内容 | 例 | 条件 |
|---------|-----|------|
| 一時的エラーの再試行 | ネットワークタイムアウト、ファイルロック | 最大2回まで |
| ツール実行の再試行 | コマンド実行の一時的失敗 | 同一コマンドの再実行、最大2回 |
| テスト再実行 | フレーキーテストの疑い | 同一テストの再実行、最大1回 |

---

## 親がユーザーにエスカレーションすべき判断レベル

以下の判断は、子から親にエスカレーションされ、親がユーザーに確認する。ユーザーの承認なしに実行してはならない。

### ファイル削除・データ破壊

| 判断内容 | 例 | リスク |
|---------|-----|-------|
| ファイル削除 | ソースファイル、設定ファイルの削除 | 復元困難 |
| データベース変更 | テーブル削除、カラム削除 | データ喪失 |
| 設定の不可逆変更 | 環境変数の削除、認証設定の変更 | サービス停止の可能性 |
| Git履歴の変更 | rebase、force push | 履歴の喪失 |

### 仕様変更・API契約の変更

| 判断内容 | 例 | リスク |
|---------|-----|-------|
| APIエンドポイントの変更 | パス、メソッド、レスポンス形式の変更 | 既存クライアントへの影響 |
| データモデルの変更 | スキーマ変更、フィールドの型変更 | データ互換性の破壊 |
| インターフェースの変更 | 関数シグネチャ、クラス構造の変更 | 依存コンポーネントへの影響 |
| 動作仕様の変更 | ビジネスロジックの変更 | ユーザー期待との乖離 |

### アーキテクチャ変更

| 判断内容 | 例 | リスク |
|---------|-----|-------|
| フレームワーク変更 | ランタイム、ビルドツールの変更 | 大規模な影響 |
| ディレクトリ構造の変更 | プロジェクト構成の再編 | 多数のファイルへの影響 |
| 依存関係の大幅変更 | メジャーバージョンアップ、ライブラリ置換 | 互換性リスク |
| デプロイ構成の変更 | CI/CD パイプラインの変更 | 運用への影響 |

### 要件の曖昧さ

| 判断内容 | 例 | リスク |
|---------|-----|-------|
| 受入基準が不明確 | 「使いやすくする」等の抽象的な要件 | 期待との乖離 |
| 複数の解釈が可能 | 要件文の曖昧な表現 | 誤った実装 |
| 前提条件の不足 | 環境情報、外部サービスの仕様が不明 | 動作不能 |
| スコープの不明確さ | どこまで対応すべきか判断できない | 過剰・過少な実装 |

## エスカレーション時のフォーマット

### 子→親へのエスカレーション

子が親にエスカレーションする際は、以下のフォーマットで報告する:

```markdown
## Escalation Request
task_id: TASK-XXX
type: <approval_needed | risk_warning | blocker>
severity: <high | medium | low>

## 状況
<現在の作業状況と、エスカレーションに至った経緯>

## 判断が必要な内容
<具体的に何を判断してほしいか>

## 選択肢
1. <選択肢A>: <メリット / デメリット>
2. <選択肢B>: <メリット / デメリット>
3. <選択肢C>: <メリット / デメリット>（あれば）

## 推奨
<子としての推奨と理由>

## 影響範囲
<この判断が影響するファイル・機能・タスク>
```

### 親→ユーザーへのエスカレーション

親がユーザーにエスカレーションする際は、子からの情報を整理して伝える:

```text
[orchestrating-agents] エスカレーション: TASK-XXX

<状況の要約>

判断が必要です:
<判断内容>

選択肢:
1. <選択肢A>（推奨: <理由>）
2. <選択肢B>
3. <選択肢C>（あれば）

影響範囲: <影響するファイル・機能>
```

## エスカレーションの応答待ちフロー

### 子→親のエスカレーション

```text
1. 子がエスカレーションを親に送信
2. 子は当該タスクを一時停止（BLOCKED状態）
3. 子は他の独立したタスクがあれば並行して進行可能
4. 親からの回答を受信したら、当該タスクを再開
```

### 親→ユーザーのエスカレーション

```text
1. 親がエスカレーションをユーザーに表示
2. 親は他のキュータスクの処理を継続可能
3. ユーザーの回答を受信
4. 親が子に SendMessage で回答を伝達
5. 子が当該タスクを再開
```

### タイムアウト

- 子→親のエスカレーション: タイムアウトなし（親は常に応答可能）
- 親→ユーザーのエスカレーション: タイムアウトなし（ユーザーの応答を待つ）
- エスカレーション待ちの間、他のタスクの処理は継続する

## 判断レベルの境界ケース

判断に迷う場合のガイドライン:

| 状況 | 判断 |
|------|------|
| 「たぶん大丈夫」と思う変更 | エスカレーションする |
| 既存コードのパターンに従った変更 | 代理承認可能 |
| 1ファイルに閉じた変更 | 代理承認可能（削除を除く） |
| 複数ファイルにまたがる構造変更 | エスカレーションする |
| テストが全てパスしている変更 | 代理承認可能（仕様変更を除く） |
| テストがない領域の変更 | エスカレーションする |

**原則: 迷ったらエスカレーションする。不要なエスカレーションのコストは、誤った自律判断のコストより遥かに小さい。**
