## 概要

親（Director）はタスクをFIFOキューで管理し、順次実行する。TaskCreate / TaskList を使ってタスクの登録・取得を行い、`task_ledger.md` と同期を保つ。`run_in_background: true` で子を起動するため、親はキュー処理中もユーザーからの追加指示を受付可能。

---

## キューの基本動作

### タスク登録

```text
1. 親がユーザーの指示をタスクに分解
2. TaskCreate で各タスクを pending 状態で登録
3. task_ledger.md にタスクを記録
```

TaskCreate で登録するタスクには以下の情報を含める:

| フィールド | 内容 |
|-----------|------|
| content | `[TASK-XXX] タスク概要` |
| status | `pending` |

### タスク取得と実行

```text
1. TaskList で pending 状態のタスクを取得
2. 最古（最初に登録された）タスクを選択
3. TaskUpdate でステータスを in_progress に変更
4. 子セッションを起動して実行を委譲
5. task_ledger.md を更新（IN_PROGRESS）
```

### タスク完了

```text
1. 子から完了報告を受信
2. TaskUpdate でステータスを completed に変更
3. task_ledger.md を更新（DONE）
4. 次の pending タスクを取得して実行（キューが空になるまで繰り返す）
```

## ユーザーの追加指示

親は `run_in_background: true` で子を起動するため、子の実行中もユーザーからの入力を受け付けられる。

### 追加指示の処理フロー

```text
1. ユーザーが新しい指示を親に伝える
2. 親が指示をタスクに分解
3. TaskCreate でキュー末尾に追加（pending 状態）
4. task_ledger.md に新タスクを追記
5. 進行中のタスクには影響を与えない（キューの末尾に追加されるだけ）
```

### 注意事項

- 追加指示が進行中タスクの方針変更である場合は、キュー追加ではなく軌道変更メカニズム（`orchestrating-agents/references/course_correction_ja.md`）を使用する
- 追加指示が進行中タスクのキャンセルを含む場合は、先にキャンセル処理を行い、その後に新タスクをキューに追加する

## task_ledger.md との同期ルール

`task_ledger.md` はユーザーが常時確認可能なタスク台帳であり、TaskCreate/TaskList の状態と常に同期させる。

### 同期タイミング

| イベント | task_ledger.md の更新 | TaskCreate/TaskUpdate |
|---------|----------------------|-----------------------|
| タスク分解時 | 新タスクを追記 | TaskCreate で登録 |
| タスク開始時 | ステータスを IN_PROGRESS に変更 | TaskUpdate で in_progress に変更 |
| タスク完了時 | ステータスを DONE に変更 | TaskUpdate で completed に変更 |
| タスク失敗時 | ステータスを FAILED に変更、理由を記録 | TaskUpdate で対応 |
| タスクブロック時 | ステータスを BLOCKED に変更、理由を記録 | TaskUpdate で pending に変更（[BLOCKED] 付記） |
| ユーザー追加指示時 | 新タスクを末尾に追記 | TaskCreate で登録 |
| 軌道変更時 | 軌道変更履歴セクションに記録 | 必要に応じてタスクを更新 |

### ステータスマッピング

| task_ledger.md | TaskCreate/TaskList |
|---------------|---------------------|
| TODO | pending |
| IN_PROGRESS | in_progress |
| DONE | completed |
| BLOCKED | pending（[BLOCKED] 付記） |
| FAILED | pending（[FAILED] 付記） |

## 依存関係の解析と実行順序決定

### 依存関係の種類

| 種類 | 説明 | 例 |
|------|------|-----|
| データ依存 | タスクAの出力がタスクBの入力 | 要件定義 → 設計 |
| 順序依存 | タスクAが完了してからタスクBを開始 | DB作成 → テーブル作成 |
| リソース依存 | 同じファイルを変更する | 同一モジュールの異なる機能実装 |

### 依存関係の記録

task_ledger.md の各タスクに `depends_on` フィールドを記録する:

```markdown
| ID | タスク | ステータス | 依存先 | 担当 |
|----|--------|-----------|--------|------|
| TASK-001 | API設計 | DONE | - | child-design |
| TASK-002 | DB設計 | DONE | - | child-design |
| TASK-003 | API実装 | IN_PROGRESS | TASK-001 | child-impl-api |
| TASK-004 | DB実装 | IN_PROGRESS | TASK-002 | child-impl-db |
| TASK-005 | 結合テスト | TODO | TASK-003, TASK-004 | - |
```

### 実行順序の決定アルゴリズム

```text
1. 依存先のないタスク（ルートタスク）を特定
2. ルートタスクをキューの先頭に配置
3. 依存先が全て DONE のタスクを実行可能として次にキューイング
4. 依存先に未完了タスクがあるものは BLOCKED としてキューに入れない
5. タスク完了時に依存先を更新し、新たに実行可能になったタスクをキューに追加
```

## 並列実行可能なタスクのグループ化

### グループ化の判定

以下の条件を全て満たすタスクは並列実行グループとしてまとめる:

1. 相互に依存関係がない
2. 対象ファイルセットが重複しない
3. 同時に実行可能な状態（依存先が全て完了）

### グループ化の実行

```text
1. 実行可能なタスク一覧を取得
2. 相互依存関係とファイルセット重複をチェック
3. 並列実行可能なタスクをグループにまとめる
4. グループ内の全タスクを同時に子セッションとして起動
   - 各子は isolation: "worktree" で起動
5. グループ内の全タスクが完了したら、次のグループへ
```

### 並列実行数の制限

- 同時に起動する子セッションは最大3つまでを推奨
- 孫セッションを含む場合、全エージェント数は最大5つまでを推奨
- リソース制約がある場合は、親が優先度に基づいて実行順序を調整する

## キュー操作のまとめ

| 操作 | 使用API | 条件 |
|------|---------|------|
| タスク登録 | TaskCreate | 親がタスク分解時 |
| 次タスク取得 | TaskList（pending, 最古） | 進行中タスク完了後 |
| ステータス更新 | TaskUpdate | タスク開始/完了/失敗時 |
| キュー追加 | TaskCreate | ユーザー追加指示時 |
| キュー確認 | TaskList | 任意のタイミングで状態確認 |
