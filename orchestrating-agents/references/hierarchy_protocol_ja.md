# 3階層プロトコル詳細仕様

## 概要

親（Director）、子（Manager）、孫（Worker）の3階層エージェント構造により、複雑なタスクを自律的に分解・並列実行する。本ドキュメントでは各階層の責務、Agent tool の呼び出し仕様、適応的階層選択、情報伝搬ルール、Git Worktree による並列実行の競合防止について詳述する。

---

## 各階層の役割と責務

### 親（Director）

ユーザーとの唯一の対話窓口であり、全体の司令塔。

**責務:**

- ユーザーの指示を受け取り、解釈・分解する
- `.orchestrating-agents/mission.md` にミッションを記録する
- タスクを TaskCreate で登録し、FIFOキューで管理する
- 子セッションを Agent tool で起動・監視する
- 子からの報告を受け取り、達成度を評価する
- エスカレーションをユーザーに転送する
- `task_ledger.md` と `mission.md` を更新する（これらは親のみが書き込む）
- Git Worktree のマージコンフリクトを解決する

**禁止事項:**

- 実装・分析等の実務を直接行わない
- 孫を直接起動しない（必ず子を経由する）

### 子（Manager）

フェーズの管理者。親から委譲されたタスクを管理・実行する。

**責務:**

- 親からの指示を受け取り、ワークオーダーを作成する
- 単純タスクの場合は自ら実務を実行する（2階層動作）
- 複雑タスクの場合は孫に分解・委譲する（3階層動作）
- 孫の進捗を監視し、品質を評価する
- 代理承認の範囲内で自律的に判断する
- 代理承認の範囲外の判断は親にエスカレーションする
- 親に完了報告・エスカレーション報告を送信する

**Agent tool の型:** `general-purpose`（Agent tool を含む全ツールにアクセス可能）

### 孫（Worker）

実務の実行者。単一タスクに集中する。

**責務:**

- 子からの指示に基づき、単一の作業を実行する
- 実装、分析、ドキュメント作成等の具体的な成果物を生成する
- 完了後、子に結果を報告する

**Agent tool の型:** 用途に応じた型（`task-executing` 等）。Agent tool は不要なため、特化型で起動する。

---

## Agent tool 呼び出しの具体的なパラメータ

### 親が子を起動する場合

```text
Agent tool(
  subagent_type: "general-purpose",
  name: "child-<task-type>",
  run_in_background: true,
  isolation: "worktree",          # 並列実行時のみ
  prompt: <親→子 指示フォーマット>
)
```

**パラメータ詳細:**

| パラメータ | 値 | 説明 |
|-----------|-----|------|
| `subagent_type` | `"general-purpose"` | 子は全ツール（Agent tool 含む）にアクセスする必要があるため必須 |
| `name` | `"child-<task-type>"` | 軌道変更時の SendMessage 宛先として使用。一意の名前を付ける |
| `run_in_background` | `true` | 親がユーザー入力を継続受付するために必須 |
| `isolation` | `"worktree"` | 並列実行時に指定。単一子のみの場合は省略可能 |
| `prompt` | 指示テンプレート | `assets/templates/parent_prompt_template_ja.md` に準拠 |

### 子が孫を起動する場合

```text
Agent tool(
  subagent_type: "<task-type>",
  name: "worker-<work-id>",
  run_in_background: true,        # 複数孫を並列実行する場合
  isolation: "worktree",          # 並列実行時のみ
  prompt: <子→孫 指示フォーマット>
)
```

**パラメータ詳細:**

| パラメータ | 値 | 説明 |
|-----------|-----|------|
| `subagent_type` | 用途に応じた型 | `task-executing`, `code-review` 等。Agent tool を含まない特化型 |
| `name` | `"worker-<work-id>"` | 子が孫を識別するための名前 |
| `run_in_background` | `true` / `false` | 複数孫の並列実行時は `true`、単一孫なら `false` でも可 |
| `isolation` | `"worktree"` | 並列実行時に指定 |
| `prompt` | 指示テンプレート | `assets/templates/child_prompt_template_ja.md` に準拠 |

## 適応的階層選択の判定ロジック

全てのタスクで3階層を使う必要はない。子は以下の判定基準に基づき、2階層と3階層を切り替える。

### 判定フロー

```text
子がタスクを受け取る
  │
  ├─ 並列実行可能な独立タスクが2つ以上あるか?
  │   ├─ YES → 3階層動作（孫を起動）
  │   └─ NO ─┐
  │           │
  ├─ タスクが異なるファイルセットを対象としているか?
  │   ├─ YES → 3階層動作（孫を起動）
  │   └─ NO ─┐
  │           │
  ├─ タスク間に依存関係がないか?
  │   ├─ YES（依存なし）→ 3階層動作（孫を起動）
  │   └─ NO（依存あり）─┐
  │                     │
  └─ 単一の逐次作業か? ──→ 2階層動作（子が直接実行）
```

### 2階層動作（子が直接実務）

**適用条件:**

- 単一タスクで分解不要
- 並列化の利点がない
- 作業量が小さい（1ファイル程度の変更）

**動作:**

子が孫を起動せず、自ら実装・分析等を実行する。完了後、親に報告する。

### 3階層動作（子が孫を起動）

**適用条件:**

- 2つ以上の独立した作業に分解可能
- 異なるファイルセットを対象とする作業がある
- 並列実行により時間短縮が見込める

**動作:**

子が作業を分解し、各孫に委譲する。孫の完了を待ち、成果物を統合して親に報告する。

## 情報伝搬ルール

### 親→子への情報伝搬

親は子に以下の情報を prompt で渡す:

1. **タスクID**: `TASK-XXX` 形式
2. **タスク種別**: 実行するスキルの種別
3. **目的と期待成果物**: 何を達成すべきか
4. **コンテキスト**: プロジェクト情報、前フェーズの成果物パス
5. **制約条件**: エスカレーション基準、代理承認範囲
6. **ワークオーダーパス**: `.orchestrating-agents/workorders/TASK-XXX.md`

### 子→孫への情報伝搬

子は孫に以下の情報を prompt で渡す:

1. **作業ID**: `TASK-XXX-W01` 形式
2. **作業内容**: 具体的な実行手順
3. **対象ファイル**: 操作対象のファイルパス一覧
4. **成果物**: 期待する出力
5. **制約条件**: スコープ外の作業の禁止事項

### 子→親への報告ルール

子は以下のタイミングで親に報告する:

1. **タスク完了時**: 成果物概要、作成ファイルリスト、品質評価
2. **エスカレーション発生時**: 判断種別、内容、選択肢（あれば）
3. **エラー発生時**: エラー内容、リトライ状況、対処方針
4. **進捗報告**: 長時間タスクの場合、中間報告

報告フォーマットは `assets/templates/report_template_ja.md` に準拠する。

## Git Worktree による並列実行の競合防止

### isolation: "worktree" の使い方

Agent tool に `isolation: "worktree"` を指定すると、そのエージェントセッションは独立した Git Worktree 上で動作する。これにより、複数のエージェントが同時にファイルを変更しても競合しない。

```text
親セッション（メインブランチ: main）
├── 子1: worktree-child-requirements（ブランチ: child-requirements）
├── 子2: worktree-child-task-exec（ブランチ: child-task-exec）
│   ├── 孫2a: worktree-worker-impl-api（ブランチ: worker-impl-api）
│   └── 孫2b: worktree-worker-impl-ui（ブランチ: worker-impl-ui）
```

### 各 Worktree は独立ブランチで動作

- 子・孫の起動時に自動的に新しいブランチが作成される
- ブランチ名は Agent tool の `name` パラメータから自動生成される
- 各ブランチは起動元のブランチから分岐する

### マージフロー（孫→子→親）

```text
1. 孫が作業を完了し、worktree ブランチにコミット
2. 孫が子に完了報告
3. 子が孫の worktree ブランチを自身のブランチにマージ
4. 全孫の完了後、子が親に完了報告
5. 親が子の worktree ブランチをメインブランチにマージ
```

### コンフリクト解決

- **孫→子のマージ**: 子が解決を試みる。解決できない場合は親にエスカレーション
- **子→親のマージ**: 親が解決する。解決に判断が必要な場合はユーザーにエスカレーション
- コンフリクト解決時は、双方の変更意図を理解した上で統合する
- 機械的な解決が困難な場合は、必ずユーザーに確認する
