# スキルカタログ

利用可能なすべてのスキルの一覧と使い方です。

## SDD（ソフトウェア設計ドキュメント）スキル

要件定義から実装まで一貫したSDDワークフローを提供するスキル群です。

### sdd-documentation（オーケストレーター）

SDDドキュメント全体を統括するオーケストレータースキルです。6つのサブスキルを統合管理し、要件定義から実装までの全工程をカバーします。

**使用開始:**

```text
docsディレクトリを初期化してください
```

**サブスキル構成:**

| サブスキル | 役割 | 成果物 |
|-----------|------|--------|
| requirements-defining | EARS記法による要件定義 | `docs/sdd/requirements/` |
| software-designing | 技術アーキテクチャ設計 | `docs/sdd/design/` |
| task-planning | AIエージェント向けタスク分解 | `docs/sdd/tasks/` |
| task-executing | タスク実行・逆順レビュー | 実装コード |
| sdd-troubleshooting | 問題分析・修正方針策定 | `docs/sdd/troubleshooting/`, `docs/sdd/tasks/` |
| sdd-document-management | ドキュメント管理・メンテナンス | `docs/sdd/archive/`, `docs/sdd/reports/` |

**主な機能:**

- EARS記法による構造化された要件定義
- 技術設計の文書化（Mermaidダイアグラム対応）
- AIエージェント向けに最適化されたタスク分解
- タスク実行と逆順レビュー（実装→タスク→設計→要件の整合性チェック）
- エージェントチームによる並列実行
- TodoWriteとのタスク同期

詳しいワークフローは[SDDワークフロー](sdd-workflow.md)を参照してください。

---

### requirements-defining

EARS記法を用いた要件定義書を作成・管理するスキルです。

**使用開始:**

```text
要件定義を作成してください
```

**EARS記法の基本パターン:**

| パターン | 形式 | 使用場面 |
|---------|------|----------|
| 基本 | `システムは〜しなければならない` | 常時適用される要件 |
| イベント | `〜の時、システムは〜しなければならない` | イベント駆動要件 |
| 条件 | `もし〜ならば、システムは〜しなければならない` | 状態依存要件 |
| 継続 | `〜の間、システムは〜しなければならない` | 継続的要件 |
| 場所 | `〜において、システムは〜しなければならない` | コンテキスト固有要件 |

### software-designing

技術アーキテクチャ・設計書を作成・管理するスキルです。

**使用開始:**

```text
設計書を作成してください
```

**作成されるドキュメント:**

- `docs/sdd/design/index.md` - 目次・アーキテクチャ概要
- `docs/sdd/design/components/*.md` - コンポーネント詳細
- `docs/sdd/design/api/*.md` - API設計詳細
- `docs/sdd/design/database/schema.md` - データベーススキーマ
- `docs/sdd/design/decisions/DEC-XXX.md` - 技術的決定事項

### task-planning

AIエージェント向け実装タスクを作成・管理するスキルです。

**使用開始:**

```text
タスクを計画してください
```

**タスクの特徴:**

- AIエージェント作業時間で20-40分程度の粒度に分解
- 各タスクに受入基準・依存関係・見積もり時間を記載
- ステータス管理: `TODO` → `IN_PROGRESS` → `DONE`
- 情報の明確性セクション（推測を排除）

---

### task-executing

タスクを実行し、実装を行うスキルです。

**使用開始:**

```text
次のタスクを実行してください
```

または

```text
docs/sdd/tasks/のタスクを進めてください
```

**主な機能:**

- `docs/sdd/tasks/`からのタスク自動読み取り
- サブエージェントを使用した自動実装
- 並列実行可能なタスクの並列処理（3つ以上でエージェントチーム使用）
- タスクごとの自動コミット作成
- タスク完了前チェックリスト（UI変更、外部連携、データ操作等）
- 逆順レビュー（実装→タスク→設計→要件）

---

### sdd-troubleshooting

動作不良や問題を分析し、修正方針を策定するスキルです。

**使用開始:**

```text
このエラーを分析してください
```

**分析フロー:**

1. 問題事象の確認（エラーメッセージ、スタックトレース、再現手順）
2. 根本原因の分析（コードを追跡して原因を特定）
3. 仕様との照合（要件定義・設計との乖離を確認）
4. 修正方針の策定（どう直すか、影響範囲は）
5. **ユーザー承認（必須ゲート）**
6. タスク分割して`docs/sdd/tasks/`に追加

---

### sdd-document-management

ドキュメントのメンテナンス・整理を行うスキルです。

**4つの機能:**

| 機能 | 説明 |
|------|------|
| 整合性チェック | requirements ↔ design ↔ tasks間の矛盾検出 |
| 実装同期チェック | 実装コードとドキュメントの乖離検出 |
| アーカイブ | 完了タスク・古い決定事項の整理 |
| ファイル最適化 | 肥大化ファイルの分割・重複統合 |

すべての操作でユーザー承認が必須です。

## 分析・インタビュースキル

### incident-rca

インシデント調査で根本原因を特定するためのなぜなぜ分析ファシリテーションスキルです。

**使用開始:**

```text
インシデントの根本原因を分析したい
```

または

```text
なぜなぜ分析を始めたい
```

**主な機能:**

- なぜなぜ分析による体系的な根本原因特定
- ユーザーの発言を漏らさず記録
- リアルタイムマインドツリー（Mermaid形式）の作成と可視化
- ファシリテーターとしての中立的なサポート（推測や意見を言わない）
- 複合要因の分割と個別分析
- 最低1つの根本原因の特定保証

**分析フロー:**

1. 成果物の保存先を確認
2. インシデントの確認（現象、日時、場所）
3. マインドツリーの初期化
4. 「なぜ？」を繰り返して根本原因に到達
5. 分析サマリーの作成

**成果物:**

- `result.md` - 分析結果（根本原因、再発防止策、マインドツリー）
- `log.md` - 会話ログ（分析プロセスの完全記録）

## ユーティリティスキル

### operations-design

運用設計コンサルタントとして、ITIL 4・SRE・DevOpsのベストプラクティスに基づいた運用設計書を作成するスキルです。

**使用開始:**

```text
運用設計書を作成したい
```

**主な機能:**

- 対象業界の最新トレンド調査（Web検索）
- AskUserQuestionツールによる体系的なヒアリング
- 会話ログによるコンテキスト保持
- インフラパターン別テンプレート（クラウドネイティブ/IaaS/オンプレミス）
- 包括的な運用設計書の作成（20セクション）
- 設計内容の一貫性チェックと客観的レビュー

**対応業界:**

Webサービス/SaaS、EC/マーケットプレイス、金融サービス、メディア/コンテンツ配信、ヘルスケア、ゲーム、IoT/組み込み、エンタープライズ

### report-summarizing

調査結果や分析レポートをエグゼクティブサマリー + 本編の構成で整理するスキルです。

**使用開始:**

```text
このドキュメントをエグゼクティブサマリー形式に変換してください
```

**4層の情報階層:**

| 層 | 対象読者 | 情報量 |
|---|---------|--------|
| エグゼクティブサマリー | 意思決定者 | 1-2ページ |
| 概要 | 関係者 | 各0.5-1ページ |
| 詳細分析 | 実装担当者 | 必要に応じて |
| Appendix | 開発者 | 必要に応じて |

**主な機能:**

- 4層構造（サマリー→概要→詳細→Appendix）による段階的情報開示
- Mermaidダイアグラムによる全体像の可視化
- GitHub/VSCodeで直接レンダリング可能なMarkdown形式

---

### jules-cli

`docs/sdd/tasks/`に記載されたタスクをGoogleの非同期コーディングエージェントJulesに依頼・管理するスキルです。

**使用開始:**

```text
次のタスクをJulesに依頼してください
```

**主な機能:**

- `docs/sdd/tasks/`からタスクを読み取り、選択したタスクをJulesに依頼
- 日本語でフォーマットされた依頼文の自動生成
- Julesの進捗状況を追跡・管理
- 完了したタスクの`docs/sdd/tasks/`自動更新
- 複数タスクの並行依頼サポート

**前提条件:**

- Jules CLIがインストールされている必要がある
- `jules auth login`で認証が完了している必要がある

### ai-code-review

指定されたPull Requestをセキュリティ、ドキュメントとの乖離、可読性・複雑度、ライブラリ選定、PR説明の適切性、既知脆弱性の検出の6観点で体系的にレビューし、PRにコメントを投稿するスキルです。

**使用開始:**

```text
PR #123 をレビューしてください
```

または再レビュー:

```text
PR #123 を再レビューしてください
```

**6つのレビュー観点:**

| 観点 | チェック内容 |
|------|-------------|
| セキュリティ | OWASP Top 10、認証・認可、入力検証、秘密情報漏洩 |
| ドキュメント乖離 | API仕様・README・設計書と実装の不一致 |
| 可読性・複雑度 | 命名、関数設計、ネスト深度、循環的複雑度 |
| ライブラリ選定 | ライセンス、メンテナンス状況、セキュリティ履歴、代替手段 |
| PR説明の適切性 | PRタイトル・説明文と変更内容の整合性 |
| 既知脆弱性の検出 | CVE・セキュリティアドバイザリに報告されたバージョンの使用 |

**主な機能:**

- 6観点の体系的なコードレビュー
- 人間向け・AI向けの2形式でレビュー指摘を併記
- PRサマリーコメント + インラインコメントの投稿
- 再レビュー時の前回指摘修正確認（resolved/unresolved/regressed等）
- 指摘の重大度分類（critical/warning/suggestion/nitpick）
- 大規模PRでのエージェントチームによる並列レビュー
