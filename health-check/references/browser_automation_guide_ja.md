# ブラウザ自動化による監視ダッシュボード調査ガイド

ブラウザ自動化ツール（Chrome DevTools MCP、Playwright、Puppeteer、Selenium等）を使って監視ダッシュボードを調査する際のガイドラインです。調査品質の保証、データ抽出の注意点、SPA固有の制約を記載しています。

---

## 1. 調査品質の保証 -- チェックリスト方式

### 原則: テキスト抽出のみで調査を完了としない

テキスト抽出（`evaluate_script`による`innerText`取得等）はデータ取得の補助手段であり、グラフの視覚的確認を省略する理由にはならない。

#### 理由

テキスト抽出ではグラフの形状（緩やかな悪化傾向、周期パターンの崩れ、散発スパイク等）は取得できない。アラート閾値に達していない「予兆」レベルの異常は、グラフを目視しなければ検出できない。

#### 必須手順

1. **各セクションをスクロールして画面に表示し、グラフの視覚的確認を行う**（スパイク、トレンド変化、パターン崩れ）
2. **テキスト抽出はアラート発火履歴やテーブルデータの補助手段**としてのみ使用する
3. **折りたたまれたセクションは展開し、データの読み込み完了を待ってから確認する**

### チェックリスト方式

調査の網羅性を保証するため、チェックリスト方式を採用する:

1. **調査開始時**にチェックリスト（URL x セクションのマトリクス）を作成する
2. **各セクションを確認するたび**にチェックを入れ、確認結果を記録する
3. **全チェックボックスが埋まるまで**レポート作成に進まない
4. **「要展開」セクション**は実際に展開操作を行ったことを明記する

```text
チェックリスト例:
- [ ] URL-1 / コスト概要 → （確認結果をここに記録）
- [ ] URL-1 / アラート状況 → ...
- [ ] URL-2 / コスト概要 → ...
...
```

#### チェックリスト運用の必須ルール

チェックリストを「作っただけで使わない」事態を防ぐため、以下のルールを遵守する:

1. **リアルタイム更新**: セクションの確認が完了したら**その場で**チェックを入れ、確認結果を記録する。後でまとめて記入しない
2. **未記入項目の禁止**: チェックリストの項目に「確認結果なし」の空欄を残さない。確認できなかった場合は理由を記録する（例: 「データ読み込み失敗」「該当セクションなし」）
3. **URL移動前の検証**: 次のURLに移動する前に、現在のURLの全チェック項目が埋まっていることを確認する。未記入項目がある場合は戻って確認する
4. **チェックリストの削除禁止**: 調査途中でチェックリストを削除しない。レポート作成完了まで維持する

## 2. スクロール制御ルール

### 原則: 小刻みスクロールで全セクションを視認する

大きなスクロールジャンプ（3000px以上）は複数セクションを一度に飛び越え、中間セクションの見落としを引き起こす。**全てのセクションを視認するために、スクロールは小刻みに行う。**

#### スクロール量の上限

| 条件 | 最大スクロール量 | 理由 |
|------|-----------------|------|
| グラフやメトリクスが並ぶダッシュボード | **800px以下** | ウィジェット1〜2個分に相当。見落としを防ぐ |
| テキスト中心のページ | **1200px以下** | 段落を跨ぐ程度 |

- **3000px以上のスクロールジャンプは禁止**。ページ末尾への移動が必要な場合でも、800px以下のスクロールを繰り返す
- スクロール後は**画面に表示されたセクション名を確認**し、飛ばしたセクションがないことを検証する

#### スクロール手順

```text
1. 現在の画面に表示されているセクション名を確認する
2. 800px以下でスクロールする
3. スクロール後に表示されたセクション名を確認する
4. 前のステップで見えていたセクションの直後のセクションが表示されていることを検証する
   - 期待するセクションが飛ばされている場合は、逆方向にスクロールして戻る
5. 表示されたセクションのグラフ・メトリクスを確認する
6. チェックリストに結果を記録する
7. ページ末尾に到達するまで 2〜6 を繰り返す
```

#### URL切替時の品質維持

複数のURLを順番に調査する場合、**後半のURLでも前半と同じスクロール手順を遵守する**。

- URL間で調査手順を省略しない。全URLで同一の手順・スクロール量を適用する
- 時間がかかる場合は、URL単位で調査を分割してユーザーに中間報告する（1URLの調査完了ごとに報告）
- 「効率化のためにスクロールを大きくする」は禁止。品質低下の主要因となる

## 3. DOMクエリによるデータ抽出の注意点

### 汎用CSSセレクタの危険性

汎用CSSセレクタ（`td`、`[role="cell"]`等）でテーブルデータを抽出すると、**目的のテーブル以外のウィジェットのデータ混入**のリスクがある。

#### 具体的なリスク

- 汎用セレクタはページ上の**全テーブルのセル**を返す
- フィルタ条件（テンプレート変数等）が効かない別セクションのテーブル値を誤って取得する可能性がある
- 過去の失敗例: あるテーブルの数値を取得する際、同一ページ上の別セクション（フィルタ条件が適用されないセクション）にある別のテーブルの値を誤って拾い、閾値超過の誤報告を行ったケースがある

### 混入の兆候

以下の兆候が見られる場合、データ混入を疑う:

- **全URLで同一の値が出現する**: テンプレート変数でフィルタされるテーブルであれば、URLごとにフィルタ条件が異なるため値も異なるはず。全URLで同じ値が出現する場合、フィルタされない別ウィジェットのデータである可能性が高い
- **行のコンテキストと紐付けできない（"unknown"）**: 行からアカウント名やサービス名を取得できない値は、目的のテーブル以外のデータである可能性がある

### 対策

1. **スコープを絞ったセレクタを使う**: 目的のウィジェットの親要素を特定してから、その配下のセルを取得する
2. **行のコンテキストを検証する**: 取得した値が同じ行のアカウント名カラムと正しく紐付けられているかを確認する
3. **クロスバリデーション**: 閾値を超える異常値を検出した場合、必ず別のダッシュボードやビューで確認してから報告する
4. **全URLで同一値が出現する場合はデータ混入を疑い、報告前に検証する**

## 4. SPA（Single Page Application）固有の制約

監視ダッシュボード（Datadog、Grafana等）の多くはSPAで構築されており、ブラウザ自動化で特有の制約がある。

### タブ干渉問題

同一ダッシュボードIDを複数タブで同時に開くと、テンプレート変数（フィルタ条件）の共有・上書きを引き起こすことがある。

- **必ず1タブずつ順次調査する**。並行タブ方式は使用不可
- 新しいURLに移動する前に、古いタブを閉じる（`close_page` → `new_page`）

### ページ読み込みの不安定性

- `navigate_page`で空白ページ（`innerText.length`が数百文字以下）になることがある。`close_page` → `new_page`の手順で解消する場合が多い
- 多数のフィルタ条件を含むURLでは**15秒以上の待機**が必要になることがある

### グラフデータの抽出制約

- 監視ツールによっては、グラフを単一のbody-level SVGにレンダリングする場合がある（Datadogで確認済み）。この場合、個別ウィジェットからプログラム的にデータ値を直接取得できない
- ウィジェットの`textContent`からはY軸範囲とX軸ラベルが取得可能で、グラフのスケール感は把握できる
- 精密な値が必要な場合は**ホバー操作によるツールチップの確認**が必要

### 表示上の制限への対処

| メッセージ | 意味 | 対処法 |
|-----------|------|--------|
| `Series Count Exceeded` | 表示上の制限でありデータの問題ではない | ウィジェット内の「Graph anyway」等の全件表示ボタンをクリックして表示する |
| `No matching results found`（間欠的） | 特定のウィジェットで間欠的に発生する不安定性 | 同じメトリクスを表示する別のウィジェットやダッシュボードを探して確認する。ページリロードで復帰する場合もある |
| `Arf. Something went wrong.`等のエラー | ページ読み込みエラー | 現在のタブを閉じ、新しいタブで同じURLを開き直す。ブラウザのコンソールログにエラー詳細が残っている場合は記録しておく |

## 5. 異常値の報告ルール

### 原則: 疑わしい異常値は検証してから報告する

閾値を超える値を検出した場合、**別のソースで裏取り**してからレポートに記載する。特に「新規検出」の重大異常は、誤報のインパクトが大きいため慎重に検証する。

### 検証手順

1. **同じメトリクスを別のダッシュボード/ビューで確認する**（アカウント別 → クロスアカウント一括、またはその逆）
2. **値の取得方法を変える**（DOMクエリ → スクリーンショット目視、またはホバーによるツールチップ確認）
3. **時間的な整合性を確認する**（関連する他のメトリクスにも同時刻に変化があるか）

### 報告時の注意

- 検証が完了していない値は**「未確認」と明記する**か、報告から除外する
- **「どのダッシュボードの、どのウィジェットで確認した値か」**を記録する
- 複数ソースで確認した場合はその旨を記載し、信頼性を示す
