# メトリクス確認ガイド

ヘルスチェックで確認する11カテゴリのメトリクスの詳細ガイドです。各カテゴリの確認項目、判定基準、注意点を記載しています。

---

## (A) コスト系

### 確認項目

| メトリクス | 確認内容 | 判定 |
|-----------|---------|------|
| 日次コスト推移 | 直近7日間のコスト推移グラフ | 急激な変動がないか |
| 前日比差分 | 前日との差額と変化率 | 変化率が±20%以上で注意 |
| 前週比差分 | 前週同曜日との差額と変化率 | 曜日パターンからの逸脱 |
| 月次累計 | 当月累計と予測値 | 予算超過の兆候 |
| リクエスト連動性 | コストとリクエスト数の相関 | 乖離はインフラ変更の兆候 |

### 注意点

- **集計遅延**: コストデータは最大24-48時間の遅延がある。最新のデータポイントは不正確なため、判断に使用しない
- **リクエスト連動性の乖離パターン**:
  - コスト増 + リクエスト増（連動）→ 需要増による正常スケーリング
  - コスト増 + リクエスト横ばい（乖離）→ インフラ変更起因（インスタンスタイプ変更、新サービス追加等）
  - コスト減 + リクエスト横ばい → リザーブドインスタンス適用、スポット活用等

## (B) アラート系

### 確認項目

| メトリクス | 確認内容 | 判定 |
|-----------|---------|------|
| 現在のトリガー状態 | ALERT/WARN/OK | ALERTは即時確認 |
| 直近の発火履歴 | 時刻・内容・優先度 | 繰り返し発火は根本原因調査 |
| 複数アカウント同時発火 | 同時刻に複数サービスで発火 | 共通基盤障害の兆候 |

### 注意点

- **フラッピング**: 短時間にALERT/OK を繰り返す場合は閾値の見直しが必要
- **複数アカウント同時発火**: 認証基盤、共通DB、ネットワーク等の共通コンポーネントを確認
- **アラート疲れ**: 常時WARN状態のアラートは実質的に機能していない。ナレッジベースに記録して除外判断

## (C) ロードバランサー系

### 確認項目

| メトリクス | 確認内容 | 判定 |
|-----------|---------|------|
| リクエスト数 | 日中/夜間パターンの維持 | パターンの崩れは異常 |
| レスポンスタイム p95 | 95パーセンタイル値 | サービス固有の閾値で判定 |
| レスポンスタイム p99 | 99パーセンタイル値 | p95正常でもp99悪化は外れ値調査 |
| レスポンスタイム max | 最大値 | 突発的な遅延の有無 |
| Target Error Rate | ターゲットグループのエラー率 | 0%が正常 |
| 5xx率 | サーバーエラー率 | 0%が正常。0.1%以上で注意 |
| Unhealthy Host Count | 異常なホスト数 | 0が正常。1以上で即時確認 |

### 注意点

- **レスポンスタイムの閾値**: サービスによって許容値が異なる。APIは100ms以内、Webページは500ms以内等
- **Unhealthy Host**: 1台でもUnhealthyがある場合は即時調査。デプロイ中の一時的なものか、恒常的なものかを区別

## (D) CPU系

### 確認項目

| メトリクス | 確認内容 | 判定 |
|-----------|---------|------|
| アプリコンテナ CPU | コンテナのCPU使用率 | 80%以上で注意 |
| EC2/VM CPU | インスタンスのCPU使用率 | 80%以上で注意 |
| 検索エンジン CPU | ES等のCPU使用率 | 70%以上で注意 |

### 注意点

- **3系統同時上昇**: アプリ・EC2・ESのCPUが同時に上昇している場合、特定テナントへの負荷集中の可能性が高い
- **バースト型インスタンス**: t系インスタンスはCPUクレジット残高も確認（カテゴリKと連動）
- **コンテナCPU**: リクエスト/リミットの設定値に対する使用率で判断

## (E) メモリ系

### 確認項目

| メトリクス | 確認内容 | 判定 |
|-----------|---------|------|
| 空きメモリ率 | 空きメモリの割合 | 20%以下で注意、10%以下で警告 |
| メモリ使用量推移 | 使用量のトレンド | 右肩上がりはリーク疑い |
| スワップ使用量 | スワップの発生有無 | 発生していれば注意 |

### 注意点

- **JVMアプリケーション**: JVMはヒープをOSに返却しない仕様のものがある。空きメモリが少なくてもJVMヒープ内に余裕がある場合は正常
- **メモリリークの判定**: 1日以上の期間で右肩上がりの傾向が続く場合はリーク疑い。再起動後にリセットされるかどうかも確認ポイント
- **OOMKill**: コンテナ環境ではOOMKillerによるプロセス強制終了の履歴も確認

## (F) データベース系

### 確認項目

| メトリクス | 確認内容 | 判定 |
|-----------|---------|------|
| RDS CPU使用率 | データベースのCPU | 70%以上で注意 |
| 接続数 | 現在の接続数と上限 | 上限の80%以上で注意 |
| デッドロック | デッドロック発生数 | 0が正常。1以上で即時調査 |
| ストレージ使用量 | ディスク使用率 | 80%以上で注意、90%以上で警告 |
| レプリケーション遅延 | リードレプリカの遅延 | 秒単位で増加傾向は注意 |

### 注意点

- **デッドロック**: 0が正常。1回でも発生している場合は即時にクエリの調査が必要
- **接続数の余裕**: max_connectionsに対する現在の接続数の割合で判断。コネクションプール設定も考慮
- **ストレージの急増**: ログの肥大化、一時テーブルの残存、バキューム未実行等が原因の場合がある

## (G) キャッシュ系（Redis/ElastiCache）

### 確認項目

| メトリクス | 確認内容 | 判定 |
|-----------|---------|------|
| メモリ余裕度 | FreeableMemory等 | **高い方が正常**（閾値が逆） |
| Eviction | キャッシュ追い出し数 | 0が正常。発生時はメモリ不足 |
| CPU使用率 | キャッシュサーバーのCPU | 65%以上で注意（シングルスレッドの場合特に） |
| 接続数 | クライアント接続数 | 急増は異常の兆候 |

### 注意点

- **閾値の方向性**: メモリ余裕度（FreeableMemory, BytesUsedForCache等）は**高い方が正常**。低下は危険。メトリクス名によって方向が異なるため注意
- **Eviction**: メモリ不足によるキャッシュ追い出し。発生するとキャッシュヒット率が低下し、バックエンド（DB等）への負荷が増加する
- **Redisのシングルスレッド特性**: CPU使用率65%でもレイテンシが悪化する場合がある

## (H) キュー/ストリーム系（SQS/Kinesis）

### 確認項目

| メトリクス | 確認内容 | 判定 |
|-----------|---------|------|
| メッセージ滞留時間 | キュー内の最古メッセージの経過時間 | 増加傾向は処理遅延 |
| DLQメッセージ数 | Dead Letter Queueのメッセージ有無 | 0が正常。1以上で内容確認 |
| スループット | 送信/受信レート | 急激な変動は注意 |
| ApproximateNumberOfMessages | キュー内の待機メッセージ数 | 増加傾向は処理能力不足 |

### 注意点

- **DLQ**: Dead Letter Queueにメッセージがある場合は、処理失敗の原因を調査する必要がある
- **滞留時間の増加傾向**: コンシューマの処理能力不足、またはコンシューマの障害を示唆
- **Kinesisのスロットリング**: シャード数の上限に達している場合に発生

## (I) 検索エンジン系（Elasticsearch/OpenSearch）

### 確認項目

| メトリクス | 確認内容 | 判定 |
|-----------|---------|------|
| ディスク使用率 | ノードのディスク使用率 | 80%で警告、90%で危険 |
| ノードCPU | ノードのCPU使用率 | 80%以上で注意 |
| クラスタステータス | Green/Yellow/Red | Yellowは冗長性低下、Redは即時対応 |
| Pending Tasks | 保留中のタスク数 | 増加傾向は処理能力不足 |
| ドキュメント数 | インデックスのドキュメント数 | 急増はインデクシング異常 |
| インデクシング遅延 | インデクシングのレイテンシ | 増加傾向は注意 |

### 注意点

- **ディスク使用率**: Elasticsearchは90%超でシャード割り当てを拒否する。80%の時点で対策が必要
- **クラスタステータスRed**: 一部のプライマリシャードが割り当てられていない状態。データ損失のリスクがあり即時対応が必要
- **連携フォールバック**: 検索エンジンとDB等との連携でフォールバックが発生している場合は、元の連携先の問題を調査

## (J) 証明書・外形監視系

### 確認項目

| メトリクス | 確認内容 | 判定 |
|-----------|---------|------|
| SSL証明書残日数 | 証明書の有効期限までの日数 | **高い方が正常**。30日以下で注意 |
| 外形監視レスポンスタイム | エンドポイントのレスポンス | ベースラインからの悪化 |
| 外形監視エラー率 | エンドポイントのエラー率 | 0%が正常 |
| 稼働時間/SLA | サービスの稼働率 | SLA目標との比較 |

### 注意点

- **SSL証明書の自動更新**: Let's Encrypt等の自動更新を使っている場合、残日数が少なくても自動更新サイクル内であれば正常。更新サイクルを理解した上で判断
- **外形監視**: 内部メトリクスが正常でも外形監視でエラーが出る場合は、ネットワークやDNS等の外部要因を調査

## (K) アプリケーション基盤系

### 確認項目

| メトリクス | 確認内容 | 判定 |
|-----------|---------|------|
| JVM Full GC回数 | Full GCの発生頻度 | 頻発は異常 |
| GCポーズ時間 | GCによる停止時間 | 100ms以上が頻発で注意 |
| エラーログカウント | サービス別のエラー数 | ベースラインからの増加 |
| Lambda エラー | サーバーレス関数のエラー | 0が正常 |
| SESバウンス率 | メールバウンス率 | 5%以上でAWSによる送信停止リスク |
| CPUクレジット残高 | バースト型インスタンスのクレジット | **高い方が正常**。枯渇でパフォーマンス低下 |

### 注意点

- **JVM GC**: Full GCが頻発する場合はヒープサイズの不足かメモリリーク。GCポーズ時間がp95レスポンスタイムに影響する
- **SESバウンス率**: AWSはバウンス率が高いアカウントの送信を停止する。5%を超えたら即時対応
- **CPUクレジット残高**: t系インスタンスはクレジット枯渇でベースライン以下のCPU性能になる。**高い方が正常**

## 閾値方向の早見表

メトリクスによって「高い方が正常」「低い方が正常」が異なります:

| パターン | メトリクスの例 |
|----------|---------------|
| **低い方が正常** | CPU使用率、エラー率、ディスク使用率、レスポンスタイム、接続数、GC回数 |
| **高い方が正常** | キャッシュメモリ余裕度(FreeableMemory)、CPUクレジット残高、SSL証明書残日数 |
| **ゼロが正常** | スロットリング、デッドロック、5xx、Unhealthy host、Eviction、DLQメッセージ数 |
