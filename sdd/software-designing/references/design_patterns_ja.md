# 設計パターンリファレンス

## アーキテクチャパターン

| パターン | 使用場面 | メリット | デメリット |
|---------|---------|---------|-----------|
| レイヤードアーキテクチャ | 標準的なWebアプリ | 責務の分離、テスト容易性 | 層間依存関係 |
| クリーンアーキテクチャ | 複雑なドメインロジック | フレームワーク非依存 | 実装コスト高 |
| マイクロサービス | 大規模システム、独立デプロイ | スケーラビリティ | 運用複雑性 |

## コンポーネント設計パターン

| パターン | 目的 |
|---------|------|
| リポジトリパターン | データアクセスロジックの抽象化 |
| サービスパターン | ビジネスロジックのカプセル化 |
| ファクトリパターン | オブジェクト生成ロジックの分離 |

## API設計

### エラーレスポンス形式

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "入力値が不正です",
    "details": [{ "field": "email", "message": "有効なメールアドレスを入力してください" }]
  }
}
```

### ページネーション

```json
{
  "data": [...],
  "pagination": { "page": 1, "perPage": 20, "total": 100, "totalPages": 5 }
}
```

## データベース設計

正規化（1NF〜3NF）とインデックス戦略、ソフトデリート（`deleted_at`カラム）を設計時に検討する。

## セキュリティパターン

### 認証方式の選択

| 方式 | 使用場面 |
|------|---------|
| JWT | SPA、モバイルアプリ |
| セッション | 従来のWebアプリ |
| OAuth 2.0 | ソーシャルログイン |
| APIキー | 内部API、B2B |

## 状態管理パターン

複雑な状態管理にはイベントソーシングやCQRS（コマンドクエリ責務分離）の採用を検討する。
